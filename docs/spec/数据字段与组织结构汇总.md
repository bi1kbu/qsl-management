# QSL 管理系统数据字段与组织结构汇总

> 核验日期：2026-07-16
> 核验范围：`src/main/java/com/bi1kbu/qslmanagement/extension/model/*.java` 与资源命名实现

## 1. 存储组织总览

- 存储机制：Halo Extension（CRD 风格）
- API Group/Version：`qsl-management.bi1kbu.com/v1alpha1`
- 统一访问路径：`/apis/qsl-management.bi1kbu.com/v1alpha1/{plural}`
- 统一对象结构：
  - `apiVersion`：资源所属 API 版本标识
  - `kind`：资源类型名
  - `metadata.name`：资源唯一 ID（持久化主键）
  - `metadata.version`：乐观锁版本号（更新时用于并发控制）
  - `metadata.creationTimestamp`：创建时间
  - `spec`：业务字段主体
  - `status`：流程状态/派生状态字段

## 2. 资源与字段清单（含中文含义）

## 2.1 StationProfile（通信地址）

- kind/plural：`StationProfile` / `station-profiles`
- 典型资源名：`qsl-station-profile-default`（单例）

`spec` 字段：
- `myCallSign`：本台呼号
- `myName`：本台姓名（My_Name）
- `myNameEn`：本台英文姓名（My_Name_en）
- `myTelephone`：本台联系电话
- `myPostalCode`：本台邮政编码
- `myAddress`：本台通信地址
- `myAddressEn`：本台英文通信地址（My_Address_en）
- `myEmail`：本台邮箱
- `stationRemarks`：本台备注

`status` 字段：
- `lastModifiedBy`：最后修改人
- `lastModifiedAt`：最后修改时间

## 2.2 StationEquipment（本台设备）

- kind/plural：`StationEquipment` / `station-equipments`
- 资源形态：多条（每条代表一个设备条目）

`spec` 字段：
- `rigName`：设备名称/机型
- `antennas`：可选天线列表
- `powers`：可选功率列表
- `modes`：可选工作模式列表
- `remarks`：设备备注

`status` 字段：
- `enabled`：是否启用

## 2.3 StationCard（本台卡片）

- kind/plural：`StationCard` / `station-cards`
- 资源形态：多条（按卡片版本管理）

- `cardVersion`：卡片版本号；系统内置值 `不发卡` 不保存为 `StationCard`，仅作为特殊卡片版本选项，不占用库存
- `imageUrl`：卡片图像地址
- `imageMediaType`：图像 MIME 类型
- `availableInventory`：当前可用库存
- `versionTotal`：该版本卡片总量
- `sortOrder`：排序号
- `remarks`：卡片备注

`status` 字段：
- `active`：是否仅支持实体 QSL 通联卡；`true` 表示仅允许实体 QSL 卡申请、普通通联和线下换卡使用，线上换卡展示但禁选且服务端拒绝；`false/null` 表示全部业务均可使用

## 2.4 SystemSetting（系统参数）

- kind/plural：`SystemSetting` / `system-settings`
- 典型资源名：`qsl-system-setting-default`（单例）

`spec` 字段：
- `guestQueryPerMinute`：访客每分钟查询限流阈值
- `onlineExchangeRequestPolicy`：线上换卡页面收到表单的处理策略（关闭线上换卡/手动/自动通过）
- `onlineAutoApprovedRequestMailPolicy`：自动审批通过的线上换卡请求本台邮件通知策略（自动不发送/自动发送）
- `requiresExchangeReview`：历史兼容字段，未配置新策略时用于推导线上换卡表单处理策略
- `autoNotifyOnCardCreated`：创建卡片后是否自动发邮件
- `autoNotifyOnCardSent`：确认发信后是否自动发邮件
- `autoNotifyOnCardReceived`：确认收卡/送达后是否自动发邮件
- `qsoCardCreatedMailPolicy/qsoCardSentMailPolicy/qsoCardReceivedMailPolicy`：通联业务邮件发送策略
- `onlineCardCreatedMailPolicy/onlineCardSentMailPolicy/onlineCardReceivedMailPolicy/onlineExchangeReviewedMailPolicy`：线上换卡业务邮件发送策略
- `cardRecordSequence`：卡片编号序列
- `receiveRecordSequence`：收卡编号序列
- `aiEnabled`：是否启用 AI 功能
- `aiProvider`：AI 接口类型，首期为 OpenAI 兼容接口
- `aiBaseUrl`：AI 接口基础地址
- `aiModel`：AI 模型名称
- `aiSecretName`：保存 API Key 的 Halo Secret 名称
- `aiTemperature`：AI 温度参数
- `aiTimeoutSeconds`：AI 调用超时时间
- `aiMaxConcurrentRequests`：AI 第三方接口最大并发请求数，默认 1，最大 10，超过上限的请求在服务端排队
- `aiMaxInputCharacters`：AI 导入解析最大输入字符数
- `aiOnlineImportParseEnabled`：线上换卡导入是否启用 AI 解析
- `aiAddressCleanupEnabled`：地址管理是否启用 AI 地址整理
- `aiSystemPrompt`：AI 全局系统提示词，可在配置页查看、编辑并重置为内置默认值
- `aiOnlineImportPrompt`：线上换卡导入 AI 解析完整提示词，需保留 `{defaultCardVersion}`、`{mode}`、`{text}` 占位符
- `aiAddressCleanupPrompt`：地址管理 AI 地址整理完整提示词，需保留 `{rows}` 占位符
- `aiCallbookAddressPrompt`：QRZ 呼号地址获取 AI 解析完整提示词，需保留 `{provider}`、`{callSign}`、`{features}` 占位符；默认提示词明确返回 `callSign/recipientName/telephone/postalCode/address/email/confidence/message`，并要求 QRZ.COM XML 优先使用最详细地址字段 `addr1`
- `qrzComEnabled`：是否启用 QRZ.COM 地址获取
- `qrzComUsername`：QRZ.COM 用户名
- `qrzComSecretName`：保存 QRZ.COM 密码的 Halo Secret 名称
- `qrzComXmlBaseUrl`：QRZ.COM 官方 XML 接口基础地址，仅允许官方域名；查询结果优先按 `call/name_fmt/fname/name/addr1/addr2/state/country/zip/email` 等 XML 字段结构化解析，字段不足时才调用 AI 兜底
- `qrzCnEnabled`：是否启用 QRZ.CN 地址获取
- `qrzCnUsername`：QRZ.CN 用户名或备注
- `qrzCnSecretName`：保存 QRZ.CN 密码和自动刷新 Cookie 的 Halo Secret 名称
- `qrzCnLookupUrlTemplate`：QRZ.CN 呼号查询页面模板，需保留 `{callSign}` 占位符，且仅允许 QRZ.CN 官方域名；服务端识别 QRZ.CN `/call/` 页面时优先使用用户名和密码登录获取 Cookie，再按配置 URL 原样请求，并按站内表单提交呼号，兼容 `q` 与 `callsign` 字段，并对概率性返回搜索页的结果进行重试和目标呼号有效性校验；页面明确提示呼号不存在时直接返回不存在，不再调用 AI；调用 AI 前抽取主体内容和图片文件名线索；QRZ 凭据测试通过后才写入配置与 Secret
- `qrzTimeoutSeconds`：QRZ 第三方查询超时时间

`status` 字段：
- `lastModifiedBy`：最后修改人
- `lastModifiedAt`：最后修改时间

## 2.7 CardRecord（创建卡片/发信/送达/收卡/制卡签发核心数据）

- kind/plural：`CardRecord` / `card-records`
- 资源名规则：正式卡片使用 `C{序号}`，从 `C1001` 开始；“不创建卡片”占位记录使用 `NC{序号}`，从 `NC0001` 开始，不占用正式卡片编号。

`spec` 字段：
- `callSign`：对方呼号
- `cardType`：卡片类型（QSO/SWL/EYEBALL）；发卡异常时使用 `原类型（ERROR）` 标记，如 `QSO（ERROR）`、`EYEBALL（ERROR）`
- `sceneType`：业务场景（QSO/SWL/ONLINE_EYEBALL/EYEBALL）
- `cardVersion`：卡片版本
- `qsoRecordName`：关联 QSO 记录 ID
- `offlineActivityName`：关联线下活动 ID
- `addressEntryName`：绑定地址编号（地址管理中的 `metadata.name`）
- `cardDate`：卡片日期
- `cardTime`：卡片时间
- `businessRemarks`：业务备注
- `createdRemarks`：制卡备注
- `sentRemarks`：发信备注
- `receivedRemarks`：收卡/送达备注
- `publicReceiptRemarks`：公开签收备注
- `cardRemarks`：卡片备注
- `cardSent`：是否已发卡
- `cardIssued`：是否已制卡
- `envelopePrinted`：是否已打包
- `cardReceived`：是否已收卡；新建卡片默认 `false`，打印工具在该字段为空或缺失时也按 `false` 处理并勾选“请回卡片”；普通业务与审计页面不直接以该字段作为“已收”展示依据
- `receiptConfirmed`：是否已签收
- `cardIssuedAt`：制卡时间
- `sentAt`：发卡时间
- `receivedAt`：收卡时间
- `createdMailStatus`：制卡邮件发送状态
- `createdMailSentAt`：制卡邮件发送时间
- `createdMailLastError`：制卡邮件最近错误
- `sentMailStatus`：发卡邮件发送状态
- `sentMailSentAt`：发卡邮件发送时间
- `sentMailLastError`：发卡邮件最近错误
- `receivedMailStatus`：收卡邮件发送状态
- `receivedMailSentAt`：收卡邮件发送时间
- `receivedMailLastError`：收卡邮件最近错误
- `mailTargetEmail`：邮件目标地址
- `receivedRecordCodes`：历史收卡编号集合，逗号分隔；2.0.7 起保留字段但普通业务、查询、统计、导入导出和前端展示不再读写，等待 2.1.x 版本移除；`cardVersion=不发卡` 时该记录不会进入待发、已发和发卡签收统计

说明：
- 通联业务创建卡片时，`qsoRecordName` 为必填；候选列表会排除已被正式卡片或“不创建卡片”占位记录占用的通联记录。
- “不创建卡片”占位记录通过 `metadata.name=NC{序号}`、`businessRemarks=不创建卡片`、空 `cardVersion/cardDate/cardTime` 表示，卡片记录清单中不展示正式卡片编号、日期、时间和版本。
- 通联业务卡片记录清单支持编辑与删除正式卡片和占位记录；占位记录通过编辑保存转为正式卡片时，会新建 `C{序号}` 资源并删除原内部占位资源。
- 制卡签发、发信确认、送达确认流程只读取正式卡片编号为 `C{序号}` 的记录，无编号占位记录不进入待制卡、查询制卡、地址绑定、发信或送达列表。
- 收卡业务支持单条编辑与批量编辑收卡日期；修改收卡日期时更新 `receivedAt` 日期段，并同步更新通过 `ReceiveRecord.spec.outboundCardNames` 关联到该卡片的 `ReceiveRecord.spec.receivedDate/receivedAt`，不再读写 `receivedRecordCodes`。

`status` 字段：
- `flowStatus`：流程状态（如已发信、已收卡片、已制卡）；普通业务、公开查询、审计页面与统计中的“已收”展示、筛选和排序以 `flowStatus=已收卡片` 为依据；`ReceiveRecord.spec.outboundCardNames` 是状态机输入事实，只要卡片存在收卡记录关联，即必须刷新为“已收卡片”；导入时不信任外部值，导出时输出后端按状态字段和收卡关联派生后的值

## 2.10 ExchangeRequest（换卡申请）

- kind/plural：`ExchangeRequest` / `exchange-requests`
- 资源形态：多条（前台申请 + 后台审核）

`spec` 字段：
- `sceneType`：业务场景，当前线上换卡为 `ONLINE_EYEBALL`
- `callSign`：申请呼号
- `cardVersion`：申请选择的卡片版本，最多两个版本，多个版本以 `、` 分隔
- `useBureau`：是否走卡片局
- `bureauName`：卡片局名称
- `email`：邮箱
- `name`：姓名
- `telephone`：电话
- `postalCode`：邮编
- `address`：地址
- `remarks`：申请备注

说明：
- 线上换卡公开表单必须选择个人地址或卡片局地址。
- 线上换卡公开表单必须从 `StationCard` 配置中选择至少一个、最多两个卡片版本；公开候选展示卡片图案、版本号、版本总量和库存余量。
- 个人地址模式下，`name/telephone/postalCode/address` 必填，`email` 可选，`useBureau=false`。
- 卡片局地址模式下，`useBureau=true`，`bureauName/postalCode/address` 保存所选或新填卡片局信息；匿名接口不会直接新增 `BureauEntry` 主数据。

`status` 字段：
- `reviewStatus`：审核状态
- `reviewReason`：审核意见/原因
- `reviewedBy`：审核人
- `reviewedAt`：审核时间

说明：
- 线上换卡申请审核在同意或拒绝后可发送审核结果邮件，目标为 `spec.email`；未配置邮箱时邮件动作跳过但保留审计日志。
- 新建线上换卡申请资源名使用 `EX{序号}`，从 `EX0001` 开始递增。
- 线上换卡申请审核通过后不会自动生成卡片记录；审核页点击“创建卡片”后，才会把 `ExchangeRequest.spec.cardVersion` 写入 `CardRecord.spec.cardVersion`，并根据申请中的个人地址或卡片局地址自动绑定 `CardRecord.spec.addressEntryName`，默认卡片备注为“期待与您空中相遇。\nLooking forward to meeting you on the air.”。
- 审核后的修改操作允许调整 `spec` 信息与 `status.reviewStatus/reviewReason/reviewedBy/reviewedAt`，也允许删除本条 `ExchangeRequest`；删除申请不会级联删除已生成的 `CardRecord`。

## 2.11 QslCardRequest（实体 QSL 卡申请）

- kind/plural：`QslCardRequest` / `qsl-card-requests`
- 资源名规则：`QCR{四位以上递增序号}`，例如 `QCR0001`

`spec（申请字段）`：
- `callSign（呼号）`：本次申请对应的对方呼号
- `qsoItems（QSO申请项目）`：逐条保存 `qsoRecordName（QSO记录编号）` 与 `cardVersion（卡片版本）`
- `addressType（地址类型）`：`PERSONAL（个人地址）` 或 `BUREAU（卡片局）`
- `addressEntryName（地址绑定）`：个人地址或卡片局 Extension 的 `metadata.name`
- `notificationEmail（通知电子邮箱）`：本次申请固定的审核通知目标
- `remarks（申请备注）`：用户填写的补充说明
- `submittedAt（提交时间）`：申请创建时间

`status（处理状态）`：
- `reviewStatus（审核状态）`：`待处理/通过/拒绝`
- `reviewReason（审核说明）`、`reviewedBy（审核人）`、`reviewedAt（审核时间）`
- `cardCreationStatus（卡片创建状态）`：`未创建/创建中/全部成功/部分失败`
- `createdCards（逐条建卡结果）`：保存 QSO、卡片版本、卡片编号、创建状态和最近错误
- `reviewMailStatus（审核邮件状态）`、`reviewMailSentAt（审核邮件发送时间）`、`reviewMailLastError（审核邮件最近错误）`、`reviewMailTargetEmail（审核邮件目标邮箱）`

说明：
- 本模型不复用 `ExchangeRequest（线上换卡申请）`，当前只允许 `sceneType（场景类型）=QSO` 的通联记录。
- 前台不读取个人历史地址；每次重新填写后，服务端按呼号、姓名、电话、邮编、地址和邮箱完整匹配复用或创建 `AddressBookEntry（地址簿条目）`。
- 审核通过后立即为每个申请项目创建一张卡片，并把 QSO 与卡片编号映射写回 `createdCards（逐条建卡结果）`。
- 公开页面的进度状态不是持久化字段，由申请、QSO 占用和关联卡片实时计算；API 返回 `statusCode/statusText/statusTextEn`，状态码覆盖 `AVAILABLE/REJECTED_REAPPLY/REVIEW_PENDING/CARD_CREATING/CARD_CREATION_FAILED/CARD_PENDING_ISSUE/CARD_ISSUED/CARD_PACKED/CARD_SENT/CARD_SIGNED`。
- `unselectableReason（不可选择原因）` 作为兼容字段保留；“已收卡片”属于本站收卡事实，不进入实体卡申请的寄送进度。

## 3. 数据组织关系（当前实现）

- QSO 与卡片：`CardRecord.spec.qsoRecordName` 关联 `QsoRecord.metadata.name`。
- 地址与卡片：`CardRecord.spec.addressEntryName` 关联 `AddressBookEntry.metadata.name`。
- 线下活动与卡片：`CardRecord.spec.offlineActivityName` 关联 `OfflineActivity.metadata.name`。
- 呼号主线：`QsoRecord.spec.callSign`、`CardRecord.spec.callSign`、`AddressBookEntry.spec.callSign`、`ExchangeRequest.spec.callSign` 使用同一业务键（呼号）。
- 实体卡申请与 QSO：`QslCardRequest.spec.qsoItems[].qsoRecordName` 关联 `QsoRecord.metadata.name`，`QslCardRequestQsoReservation.spec.qsoRecordName` 为待处理期间的持久化唯一占用。
- 实体卡申请与地址：`QslCardRequest.spec.addressEntryName` 按 `addressType（地址类型）` 关联 `AddressBookEntry.metadata.name` 或 `BureauEntry.metadata.name`。
- 实体卡申请与卡片：`QslCardRequest.status.createdCards[]` 保存 `qsoRecordName（QSO记录编号）` 到 `cardRecordName（卡片编号）` 的一对一映射；卡片的 `businessRemarks（业务备注）` 同时保留申请编号用于幂等恢复。
- 地址与邮件：卡片邮件目标地址最终落在 `CardRecord.spec.mailTargetEmail`，来源于地址簿匹配及发送流程写回。
- 审核到卡片：`ExchangeRequest` 审核通过后不会自动生成 `CardRecord`；需在审核页点击“创建卡片”显式生成卡片记录（含 `flowStatus` 流程状态）。
- 制卡流程：制卡签发会更新 `CardRecord.spec.cardIssued/cardIssuedAt`，并将 `CardRecord.status.flowStatus` 置为“已制卡”。
- 收卡编号：送达/收卡确认维护 `ReceiveRecord.metadata.name` 与 `ReceiveRecord.spec.outboundCardNames`，允许多张收卡记录关联同一发卡记录；无发卡关联的收卡记录可在收卡记录查询中人工关联到匹配的正式发卡记录，关联后刷新目标卡片已收状态；`CardRecord.spec.receivedRecordCodes` 仅保留为历史字段。
- 审计：写操作通过 `QslAuditLog` 追加记录，`resourceType + resourceName` 指向受影响对象。

## 4. 关键枚举/取值组织（当前代码约束）

- 业务场景：`QSO` / `SWL` / `ONLINE_EYEBALL` / `EYEBALL`。
- 卡片类型：`QSO` / `SWL` / `EYEBALL`（多处校验采用该集合）。

## 5. 资源命名规则汇总

- QSO：`QSO{n}`，`n` 从 `1001` 起递增。
- 卡片：`C{n}`，`n` 从 `1001` 起递增。
- 不创建卡片占位：`NC{n}`，`n` 从 `0001` 起递增，仅后台持久化，前台卡片 ID 空白展示。
- 线上换卡申请：`EX{n}`，`n` 从 `0001` 起递增。
- 实体 QSL 卡申请：`QCR{n}`，`n` 从 `0001` 起递增。
- 实体卡申请 QSO 占用：`qcr-qso-{hash}`，由标准化 QSO 编号确定性生成。
- 地址：`{CALLSIGN}-{n}`，`n` 从 `1` 起递增，`CALLSIGN` 强制大写。
- 卡片局：`BURO-{n}`，`n` 从 `1` 起递增。
- 单例：
  - 通信地址：`qsl-station-profile-default`
  - 系统参数：`qsl-system-setting-default`
  - 数据迁移状态：`qsl-migration-state-default`
