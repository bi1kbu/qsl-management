# QSL 项目信息结构化清单

更新时间：2026-05-22

代码核验范围：`插件工程根目录`、`tools/CardPrint`

## 1. 项目定位
1. 项目名称：QSL 管理系统（Halo2 插件 + 本地打印工具）。
2. 核心目标：围绕通联、换卡、制卡签发、发信与送达确认、收卡编号管理，提供可持久化的后台业务流与前台自助页面。
3. 技术形态：
   - Halo2 插件后端：Java 21 + Spring WebFlux + Halo Extension API
   - Halo2 插件前端：Vue 3 + TypeScript
   - 本地打印工具：`tools/CardPrint`（Python）

## 2. 仓库结构
1. `src/`：Halo2 插件后端、扩展模型、资源清单与权限模板。
2. `ui/`：Halo2 控制台前端工程。
3. `tools/CardPrint/`：本地在线打印工具（独立 Python 项目）。
4. `docs/spec/`：结构化项目信息文档。
5. `docs/spec/ProductDefinition.md`：产品定义（约束性文档）。
6. `docs/spec/BackendApiContract.md`：后端 API 合同（输出性文档，需与代码同步）。
7. `agents.md`：协作与约束规则（约束性文档）。

## 3. 后端架构（src/main/java）
1. 插件启动与默认资源：
   - `QslManagementPlugin`：确保默认 `SystemSetting`、`StationProfile` 实例存在。
2. Extension 注册：
   - `QslExtensionSchemeRegistrar`：注册并维护名称索引。
3. 业务服务：
   - `QslConsoleActionService`：后台业务操作（确认、状态流转等）。
   - `QslImportExportJobService`：导入导出任务。
   - `QslNotificationMailService`：邮件发送与状态记录。
   - `QslAiService`：AI 配置测试、OpenAI 兼容协议调用、线上换卡导入 AI 解析、地址整理与 QRZ 呼号地址解析。
   - `QslQrzAddressLookupService`：QRZ.COM 官方 XML 查询、QRZ.CN 页面抓取与主体预处理、Secret 凭据写入与地址预览。
   - `QslPublicApiService`：前台公开业务处理。
4. Endpoint 分层：
   - `console.api.qsl-management.bi1kbu.com`：控制台 API
   - `api.qsl-management.bi1kbu.com`：公开 API 与前台页面路由
5. 前台渲染与短码：
   - `QslPublic*PageRenderService`：前台页面 HTML 渲染
   - `Qsl*EmbedContentTransformer`：文章/页面短码转嵌入卡片

## 4. 前端架构（ui/src）

   - `index.ts`：注册 `/qsl` 路由与菜单分组。
2. 菜单配置：
   - `constants/menu-modules.ts`：模块定义、权限节点、依赖关系。
3. 模块承载：
   - `views/MenuLayoutView.vue`：分组布局
   - `views/MenuModuleView.vue`：按模块 key 分发到具体页面组件

5. 业务页面：

## 6. 核心数据模型（Extension）
1. `SystemSetting`：系统参数（限流、审核开关、序号等）。
2. `StationProfile`：本台通信地址与联系信息。
3. `StationEquipment`：本台设备信息。
4. `StationCard`：本台卡片版本、库存、排序与 Halo 附件库图片引用；后台通过图片选择入口打开附件库弹窗，在弹窗内上传或选择图片后写入附件引用；卡片图案原图不写入本资源，旧版 `imageUrl` base64 字段不再兼容，后台不再提供旧图片字段清理入口。系统内置卡片版本“不发卡”不保存为 `StationCard`，仅在卡片版本选择器中作为特殊选项存在，库存视为无限，不显示在“配置-本台卡片”菜单中；发送链路页面隐藏该类记录，收卡业务仍显示并允许关联。
5. `QsoRecord`：通联日志。
6. `CardRecord`：卡片主记录（场景、版本、状态、备注、邮件状态、地址绑定、签收状态）。线下换卡卡片不绑定地址；线上换卡“送达确认”语义为确认签收/签收状态，签收事实为 `CardRecord.spec.receiptConfirmed`。
7. `ExchangeRequest`：线上换卡申请，状态区记录审核结果与审核通知邮件发送状态。
8. `OfflineActivity`：线下换卡活动。
9. `AddressBookEntry`：通信地址簿，保存呼号、姓名、电话、邮编、去向国、收件地址、邮箱与地址备注；去向国非必填。
10. `BureauEntry`：卡片局地址簿，保存卡片局名称、电话、邮编、去向国、地址与备注；去向国非必填。
11. `EquipmentCatalogEntry`：设备库。
12. `QslAuditLog`：审计日志。
13. `ImportExportJob`：导入导出任务。
14. `ReceiveRecord`：独立收卡事实，是收卡业务的收卡编号来源，记录收卡编号、呼号、卡片类型、来源场景、关联发卡记录与匹配状态。
15. `OfflineExchangeCard`：线下换卡活动卡，记录活动、待认领呼号、发出状态和线下备注。

导入导出备份数据集覆盖 `qso-record/card-record/receive-record/exchange-request-review/offline-activity/offline-exchange-card/address-management/bureau-management/equipment-catalog/system-setting/station-profile/station-equipment/station-card`；`all` 导出需同时包含业务数据、收卡事实、线下换卡活动卡、线下换卡活动清单与配置菜单数据。导入导出前端识别 ZIP 内 CSV 时优先按 `id#数据集` 与文件名精确匹配，避免 `station-card.csv` 被宽泛 `card` 关键词误判为 `card-record`；服务端导入 `card-record` 时跳过 `qsl-station-card-*` 或 `station-card-*` 资源名，导出 `card-record` 时过滤此类历史脏记录。2.0.0 前旧导出包需通过 `tools/convert_legacy_export_to_qsl2.py` 转换，转换时从旧卡片记录聚合提取收卡关联与线下活动卡关系，并过滤误进入 `card-record.csv` 的 `qsl-station-card-*` 本台卡片版本占位记录。导入导出页面提供“旧版本迁移”页签，用于 Halo 存储仍保留旧模型数据时原地迁移；执行前必须全量备份，并输入确认文字“确认迁移旧版本数据”。

关键新增字段事实：

1. `SystemSetting.spec` 包含 `guestQueryPerMinute/onlineExchangeRequestPolicy/onlineExchangeRequestCooldownMinutes/onlineAutoApprovedRequestMailPolicy/requiresExchangeReview/cardRecordSequence/receiveRecordSequence`，线上换卡页面收到表单的处理策略按 `DISABLED/MANUAL/AUTO_APPROVE` 保存，同呼号重复提交冷却时间默认 5 分钟且适用于待审核、已通过和已拒绝申请，系统自动审批通过后的本台邮件通知策略按 `AUTO_SKIP/AUTO_SEND` 保存，接收人为 `StationProfile.spec.myEmail`，`requiresExchangeReview` 作为历史兼容字段保留；邮件策略按业务拆分为 `qsoCardCreatedMailPolicy/qsoCardSentMailPolicy/qsoCardReceivedMailPolicy`、`onlineCardCreatedMailPolicy/onlineCardSentMailPolicy/onlineCardReceivedMailPolicy/onlineExchangeReviewedMailPolicy`，取值为 `AUTO_SKIP/MANUAL/AUTO_SEND`；业务动作触发自动策略时，`AUTO_SKIP` 只写入对应邮件状态 `SKIPPED`，不触发 Halo 通知；`offlineAutoNotifyOnCardReceived` 作为历史配置字段保留但不再使用，线下换卡收卡与送达确认默认不发送邮件；旧的全局自动通知字段和场景布尔字段仅作为已有配置读取兜底。AI 配置字段包含 `aiEnabled/aiProvider/aiBaseUrl/aiModel/aiSecretName/aiTemperature/aiTimeoutSeconds/aiMaxConcurrentRequests/aiMaxInputCharacters/aiOnlineImportParseEnabled/aiAddressCleanupEnabled/aiSystemPrompt/aiOnlineImportPrompt/aiAddressCleanupPrompt/aiCallbookAddressPrompt`，提示词为完整模板，配置页可查看、编辑并按项重置；`aiMaxConcurrentRequests` 默认 1、最大 10，用于服务端第三方 AI 调用并发排队控制；自定义提示词不允许改变系统指定 JSON 返回结构。QRZ 配置字段包含 `qrzComEnabled/qrzComUsername/qrzComSecretName/qrzComXmlBaseUrl/qrzCnEnabled/qrzCnUsername/qrzCnSecretName/qrzCnLookupUrlTemplate/qrzTimeoutSeconds`；服务端只允许 QRZ.COM / QRZ.CN 官方域名，QRZ 凭据测试通过后才写入配置与 Secret；QRZ.COM 默认同时获取官方 XML 与资料页，资料页 Biography/Detail 主体可用时以资料页正文为主、XML 为辅交给 AI，资料页不可用时才回退 XML，XML 只有国家、省份、州等宽泛行政区时不直接作为有效地址；QRZ.CN 抓取结果会在服务端优先使用用户名和密码登录获取 Cookie，再执行配置 URL 原样请求、表单查询、概率性搜索页重试、目标呼号有效性校验、编码解码、主体内容与图片文件名线索抽取；页面明确提示呼号不存在时直接返回不存在，不再调用 AI，未获取到有效目标呼号页面时明确报错。AI API Key、QRZ.COM 密码、QRZ.CN 密码与自动刷新 Cookie 写入 Halo `Secret`，不写入 `SystemSetting.spec` 明文字段。
2. `StationCard.spec` 包含库存与排序字段：`availableInventory/versionTotal/sortOrder`。
3. `QsoRecord.spec` 包含 `sceneType/myQth/operator`。
4. `CardRecord.spec` 包含 `sceneType/offlineActivityName/addressEntryName/businessRemarks/createdRemarks/sentRemarks/receivedRemarks/publicReceiptRemarks/cardIssued/envelopePrinted/receiptConfirmed/receivedRecordCodes` 以及三类邮件状态字段。其中 `receiptConfirmed` 表示线上换卡等发卡场景的确认签收事实；`receivedRecordCodes` 为待 2.1.x 版本移除的历史字段，2.0.7 起普通业务、查询、统计、导入导出和前端展示不再读写；线下换卡卡片不写入 `addressEntryName`；`cardVersion=不发卡` 作为系统内置特殊值仅用于标记无需发卡的卡片，不占用库存，也不计入待发和已发统计；线上换卡审核通过后不自动创建卡片，审核页“创建卡片”按钮和线上换卡导入创建的卡片默认 `cardRemarks` 为“期待与您空中相遇。\nLooking forward to meeting you on the air.”；通联业务创建卡片页下方待创建记录仅显示尚未创建卡片且未标记“不创建卡片”的 QSO/SWL 记录，页面默认不选中卡片版本，操作栏一键创建正式卡片前必须先人工选择卡片版本，用户可手动选择系统内置“不发卡”，业务备注为空，通联业务默认 `cardRemarks` 为“通联愉快，期待空中常见。\nNice QSO，Hope to catch you on the air often.”。
5. `ExchangeRequest.spec` 包含 `sceneType/cardVersion`；`ExchangeRequest.status` 包含 `reviewStatus/reviewReason/reviewedBy/reviewedAt`、审核通知邮件字段 `reviewMailStatus/reviewMailSentAt/reviewMailLastError/reviewMailTargetEmail`，以及建卡纠偏字段 `cardCreated/cardCreatedAt/cardCreatedBy/createdCardRecordName`。新建换卡申请资源名使用 `EX####` 递增格式。

## 7. 场景主字段约定
1. `sceneType` 使用：`QSO`、`SWL`、`ONLINE_EYEBALL`、`EYEBALL`。
2. 通联业务：主场景 `QSO/SWL`。通联日志清单支持按呼号和通联日期范围筛选，日期范围按 `QsoRecord.spec.date` 做闭区间匹配，并影响当前筛选导出范围。通联日志提供 `ADIF格式导入`、`ADIF格式导出`、`Cabrillo格式导入` 与 `Cabrillo格式导出` 页签；手工新增记录默认使用 `UTC+8`，启用 UTC 实时填充、ADIF 导入与 Cabrillo 导入保留 `UTC`；导入页不新增后端 API，解析预览后通过现有 `QsoRecord` 创建能力写入记录，无法落入现有模型的 ADIF 字段、Cabrillo 头字段、比赛交换信息和 QSO 行扩展字段追加到备注；ADIF 导入兼容字段只在原对应字段缺失时启用：`GRIDSQUARE` 回填对方位置，`MY_GRIDSQUARE` 回填本台 QTH，`TX_PWR` 写入我台发射功率且缺失时才回落到设备默认功率；ADIF 导入预览默认跳过重复记录，本站导出的 `APP_QSLMS_RECORD_ID` 已存在时按本站记录编号判重，外部 ADIF 或缺少本站记录编号时按 `sceneType/callSign/date/time/timezone/myRigMode/freq` 组成业务键判重；“导入可用记录”只创建非重复记录，“覆盖导入”需二次确认，仅批量更新已匹配到现有资源名或业务判重键的重复记录；同一业务判重键下存在多条现有记录时全部覆盖更新，文件内部重复和无效记录不写入；导出已勾选记录或当前筛选结果，不改变 `QsoRecord` 数据。ADIF 和 Cabrillo 导出日期时间均按记录时区换算为 UTC，并兼容 `UTC+8/UTC+08:00/GMT+8/北京时间/Asia/Shanghai` 等等价时区写法；ADIF 导出仅在 `FREQ` 存在 ADIF `Number` 类型的 MHz 数值时写入频率，`{BAND} BAND` 占位按 ADIF 3.1.7 波段枚举写入 `BAND`，不写入非数值 `FREQ`，由导入保留到备注中的 `MY_STATE/STATE/GRIDSQUARE/MY_GRIDSQUARE/SIG/SIG_INFO/QSO_DATE_OFF/TIME_OFF` 等字段导出时恢复为独立 ADIF 字段；ADIF 导出页内支持 `MY_SIG` 下拉、`MY_SIG_INFO` 输入，本地持久化保存导出参数，`MY_SIG` 为空时显示为“不添加”，`OPERATOR` 导出时强制采用呼号，`TX_PWR` 导出时按 ADIF `Number` 类型规范化为瓦特数值；Cabrillo 导出页支持 `CONTEST/OPERATORS/CATEGORY-*` 配置，并生成 Cabrillo 3.0 文本文件。
3. 线上换卡业务：主场景 `ONLINE_EYEBALL`。
4. 线下换卡业务：主场景 `EYEBALL`。

## 8. 接口边界（按代码当前状态）
1. 控制台 API 前缀：
   - `/apis/console.api.qsl-management.bi1kbu.com/v1alpha1/...`
2. 公开 API 前缀：
   - `/apis/api.qsl-management.bi1kbu.com/v1alpha1/...`
3. 公开页面路由（示例）：
   - 查询卡片：`/cards/page`
   - 换卡（线上）：`/ONLINE_EYEBALL`、`/ONLINE_EYEBALL/{cardId}`
   - 换卡（线下）：`/EYEBALL`、`/EYEBALL/{cardId}`
   - 签收：`/receipt-public`、`/receipt-public/{cardId}`
4. 线上换卡与线下换卡公开页面路径保持不变，但服务端 endpoint 与页面渲染服务分离实现，避免两个换卡场景的页面字段、脚本和提交逻辑互相耦合。
5. 公开提交与查询路由：
   - `GET /qso-public/records`
   - `GET /qso-public/grids`：匿名返回通联记录中的对方四位网格清单，按 `QsoRecord.spec.qth` 识别 4/6/8 位 Maidenhead 网格，统一截取四位后聚合；地区文字、地址或混合 QTH 跳过，同一网格返回去重呼号集合，`detailLevel（详情级别）` 为完整时返回通联明细，为简略时只返回呼号清单。
   - `GET /overview-public/summary`
   - `GET /exchange-online/-/bureaus`
   - `GET /exchange-online/-/station-cards`
   - `GET /exchange-offline/-/activities`
   - `POST /exchange-online/-/requests`：同呼号存在待审核线上换卡申请时返回 `409/QSL-409-0001`；同呼号上一条申请仍处于冷却期时返回 `429/QSL-429-0001`；写入成功后返回本站通信地址。
   - `POST /exchange-offline/-/confirm`
   - `POST /receipt-public/-/confirm`
6. 控制台动作路由：
   - `GET /overview/summary`
   - `GET /reports/summary`：统计报表聚合，返回汇总指标与 `charts.monthlyCardFlow` 月度收发卡片折线图数据。
   - `POST /mail-send-confirms/{cardRecordName}/confirm`
   - `POST /mail-receive-confirms/confirm`
   - `POST /mail-receive-confirms/{cardRecordName}/received-date`
   - `POST /card-mutations/{cardRecordName}/resend`：卡片重发，清理制卡、打包、发信及相关邮件状态并保留收卡事实。
   - `POST /card-mutations/{cardRecordName}/mark-error`：发卡异常，将卡片类型标记为 `原类型（ERROR）`，可追加异常备注。
   - `POST /card-mutations/{cardRecordName}/mark-resend`：标记重发，解除发卡异常；控制台随后调用重发接口清理制卡、打包、发信状态。
   - `POST /exchange-requests/{name}/approve`
   - `POST /exchange-requests/{name}/reject`
   - `POST /exchange-requests/{name}/create-card`：已通过申请显式创建或复用线上换卡卡片，并同步写入建卡状态。
   - `POST /exchange-requests/{name}/mark-card-created`：已通过申请历史纠偏标记已发卡，只写入建卡状态，不创建卡片。
   - `POST /exchange-requests/{name}/notify`：发送线上换卡审核通知，成功后写入 `ExchangeRequest.status.reviewMailStatus=SENT` 并阻止重复发送。
   - `POST /online-card-imports`：导入手工文本或 BH6SYX 卡片广场数据，服务端直接创建 `ONLINE_EYEBALL` 场景 `CardRecord`，并按需创建/绑定 `AddressBookEntry`；仅允许 `对方已寄出，待我签收`、`待双方寄出` 两类状态；“交换ID”和“对方备注”不提交和入库，后续流程统一围绕本站卡片编号处理。
   - `POST /ai-config-tests`：测试 AI 接口配置并可写入 API Key Secret。
   - `POST /ai-address-normalizations/preview`：AI 地址整理预览，不落库。
   - `POST /ai-address-normalizations/apply`：应用 AI 地址整理结果，仅更新地址簿地址字段。
   - `POST /ai-online-import-parses`：线上换卡导入文本 AI 解析，返回预览行，不直接创建卡片。

## 11. 本地打印工具（tools/CardPrint）
1. 主入口：
   - `python -m cardprint.cli ui online`
2. 当前作业流：
   - 配置登录 -> 卡片打印 -> 制卡确认 -> 打印封面（打包）-> 打包确认
   - 补打信封作为独立页签位于打包确认后，可按地址簿或卡片局主数据补打信封。
3. 与后台关系：
   - 读取 QSL 后台卡片与地址信息生成打印队列。
   - 打印状态通过人工确认页签回写，避免自动误回写。
4. 在线桥接代码事实：
   - 默认 `base_url` 为 `http://localhost:8090`，但配置可传入其他地址。
   - 卡片/信封基础数据读取 `card-records`，卡片打印会按 `CardRecord.spec.qsoRecordName` 额外读取 `qso-records` 并注入 `qsoInfo`；通联业务制卡的发射地址字段优先使用 `QsoRecord.spec.myQth`，旧预设字段 `qth` 与新预设字段 `my_qth` 均按此规则取值，缺失时才兜底到对方 `qth`；线下换卡制卡仍优先使用关联活动的 `OfflineActivity.spec.activityLocation`；打印字段映射中 `CardRecord.spec.cardReceived` 为空或缺失时按未收卡处理，默认勾选“请回卡片”。
   - 信封打印纳入未打包（`envelopePrinted=false`）且卡片版本不是“不发卡”的卡片记录；信封补全会读取 `station-profiles`、`address-book-entries`、`bureau-entries`；封面打印页提供去向类型下拉筛选，去向国为空按国内卡片处理，存在去向国内容按国际卡片处理。打包确认使用同一未打包且非“不发卡”口径回写 `envelopePrinted`。
   - 补打信封读取 `address-book-entries` 与 `bureau-entries` 生成独立队列，复用信封预设，支持按呼号、地址编号或卡片局编号筛选，支持当前行打印、勾选批量打印与全部打印，不回写 `CardRecord` 状态。
   - 地址管理的待绑定地址呼号清单只统计仍需发卡的未绑定卡片；若呼号对应的未绑定正式卡片全部为 `cardVersion=不发卡`，不显示在待绑定清单中，地址自动绑定也跳过此类卡片。卡片异动中把卡片版本从“不发卡”改回实际卡片版本时，若尚未绑定地址且存在同呼号地址簿记录，自动回填第一条同呼号地址。
   - 补打眼球卡片使用独立页签和专用预设，预设路径保存到 `bridge_config.json` 的 `presets.eyeball_reprint_card`；页面手工输入呼号与日期时间，支持实时取当前日期时间，打印行固定 `cardType=EYEBALL`、“发出卡片”状态与“请回卡片”状态，不读取或回写 `CardRecord` 状态。
   - 卡片版本列表通过公开接口 `GET /apis/api.qsl-management.bi1kbu.com/v1alpha1/exchange-online/-/station-cards` 读取，本质数据来源仍为 `station-cards`，不从 `card-records` 反推，并按 `sortOrder` 与版本号排序；本台通信地址仍通过受保护的 `station-profiles` Extension API 读取。
   - 线上换卡申请审核通过后点击“创建卡片”时，按申请地址复用或创建地址资源，并写入卡片记录的 `addressEntryName`。
   - 卡片二维码拼接支持 `qrcode.path_mappings` 短路径映射，在线打印配置页可在站点地址下方配置线下换卡、线上换卡、签收确认三类短路径；默认将公开页面长路径映射为 `/EYEBALL`、`/ONLINE_EYEBALL`、`/rp` 后再追加卡片 ID 与 `cs` 参数。
   - 登录并拉取卡片版本时，网络请求在后台线程执行，避免界面短暂未响应；只补齐空白本台通信地址字段，不覆盖已有本台姓名、电话、邮编、地址。若受保护接口返回登录页或 HTML，工具明确提示认证或权限问题，不再静默解析为空数据。
   - 回写采用 Halo Extension 完整 `PUT`，先读取当前对象再合并 `spec`；`cardIssuedAt` 使用 `yyyy-MM-dd HH:mm:ss` 文本格式；回写会同步刷新 `CardRecord.status.flowStatus`，并按后台状态流转规则补齐或清理相关状态字段。
   - 打印预设字段支持 `fixed_text` 固定文本，用于固定辅助说明、提示语或版面标识；固定文本字段由标定工具维护，打印工具和在线打印工具不要求录入该字段，预览与打印时优先输出固定文本。
