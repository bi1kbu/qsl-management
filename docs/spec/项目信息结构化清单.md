# QSL 项目信息结构化清单

更新时间：2026-05-04

代码核验范围：`qsl-plugin`、`tools/CardPrint`

## 1. 项目定位
1. 项目名称：QSL 管理系统（Halo2 插件 + 本地打印工具）。
2. 核心目标：围绕通联、换卡、制卡签发、发信与送达确认、收卡编号管理，提供可持久化的后台业务流与前台自助页面。
3. 技术形态：
   - Halo2 插件后端：Java 21 + Spring WebFlux + Halo Extension API
   - Halo2 插件前端：Vue 3 + TypeScript
   - 本地打印工具：`tools/CardPrint`（Python）

## 2. 仓库结构
1. `qsl-plugin/`：Halo2 插件工程（后端 + 控制台前端）。
2. `tools/CardPrint/`：本地在线打印工具（独立 Python 项目）。
3. `docs/spec/`：结构化项目信息文档。
4. `ProductDefinition.md`：产品定义（约束性文档）。
5. `BackendApiContract.md`：后端 API 合同（输出性文档，需与代码同步）。
6. `agents.md`：协作与约束规则（约束性文档）。

## 3. 后端架构（qsl-plugin/src/main/java）
1. 插件启动与默认资源：
   - `QslManagementPlugin`：确保默认 `SystemSetting`、`StationProfile` 实例存在。
2. Extension 注册：
   - `QslExtensionSchemeRegistrar`：注册并维护名称索引。
3. 业务服务：
   - `QslConsoleActionService`：后台业务操作（确认、状态流转等）。
   - `QslImportExportJobService`：导入导出任务。
   - `QslNotificationMailService`：邮件发送与状态记录。
   - `QslPublicApiService`：前台公开业务处理。
4. Endpoint 分层：
   - `console.api.qsl-management.halo.run`：控制台 API
   - `api.qsl-management.halo.run`：公开 API 与前台页面路由
5. 前台渲染与短码：
   - `QslPublic*PageRenderService`：前台页面 HTML 渲染
   - `Qsl*EmbedContentTransformer`：文章/页面短码转嵌入卡片

## 4. 前端架构（qsl-plugin/ui/src）
1. 入口：

2. 菜单配置：
   - `constants/menu-modules.ts`：模块定义、权限节点、依赖关系。
3. 模块承载：

5. 业务页面：
   - `views/modules/*Module.vue`（通联、卡片、签发、确认、审计、数据、配置等）。

## 6. 核心数据模型（Extension）
1. `SystemSetting`：系统参数（限流、审核开关、序号等）。
2. `StationProfile`：本台通信地址与联系信息。
3. `StationEquipment`：本台设备信息。
4. `StationCard`：本台卡片版本、库存、排序与 Halo 附件库图片引用；后台通过图片选择入口打开附件库弹窗，在弹窗内上传或选择图片后写入附件引用；卡片图案原图不写入本资源，旧版 `imageUrl` base64 字段不再兼容，后台不再提供旧图片字段清理入口。
5. `QsoRecord`：通联日志。
6. `CardRecord`：卡片主记录（场景、版本、状态、备注、邮件状态、地址绑定、收卡编号）。
7. `ExchangeRequest`：线上换卡申请，状态区记录审核结果与审核通知邮件发送状态。
8. `OfflineActivity`：线下换卡活动。
9. `AddressBookEntry`：通信地址簿。
10. `BureauEntry`：卡片局地址簿。
11. `EquipmentCatalogEntry`：设备库。
12. `QslAuditLog`：审计日志。
13. `ImportExportJob`：导入导出任务。

导入导出备份数据集覆盖 `qso-record/card-record/exchange-request-review/offline-activity/address-management/bureau-management/equipment-catalog/system-setting/station-profile/station-equipment/station-card`；`all` 导出需同时包含业务数据、线下换卡活动清单与配置菜单数据。

关键新增字段事实：

1. `SystemSetting.spec` 包含 `guestQueryPerMinute/requiresExchangeReview/cardRecordSequence/receiveRecordSequence`，邮件策略按业务拆分为 `qsoAutoNotifyOnCardCreated/qsoAutoNotifyOnCardSent/qsoAutoNotifyOnCardReceived`、`onlineAutoNotifyOnCardCreated/onlineAutoNotifyOnCardSent/onlineAutoNotifyOnCardReceived/onlineAutoNotifyOnExchangeReviewed`、`offlineAutoNotifyOnCardReceived`；旧的全局自动通知字段仅作为已有配置读取兜底。
2. `StationCard.spec` 包含库存与排序字段：`availableInventory/versionTotal/sortOrder`。
3. `QsoRecord.spec` 包含 `sceneType/myQth/operator`。
4. `CardRecord.spec` 包含 `sceneType/offlineActivityName/addressEntryName/businessRemarks/createdRemarks/sentRemarks/receivedRemarks/publicReceiptRemarks/cardIssued/envelopePrinted/receiptConfirmed/receivedRecordCodes` 以及三类邮件状态字段。
5. `ExchangeRequest.spec` 包含 `sceneType/cardVersion`；`ExchangeRequest.status` 包含 `reviewStatus/reviewReason/reviewedBy/reviewedAt` 以及审核通知邮件字段 `reviewMailStatus/reviewMailSentAt/reviewMailLastError/reviewMailTargetEmail`。

## 7. 场景主字段约定
1. `sceneType` 使用：`QSO`、`SWL`、`ONLINE_EYEBALL`、`EYEBALL`。
2. 通联业务：主场景 `QSO/SWL`。
3. 线上换卡业务：主场景 `ONLINE_EYEBALL`。
4. 线下换卡业务：主场景 `EYEBALL`。

## 8. 接口边界（按代码当前状态）
1. 控制台 API 前缀：
   - `/apis/console.api.qsl-management.halo.run/v1alpha1/...`
2. 公开 API 前缀：
   - `/apis/api.qsl-management.halo.run/v1alpha1/...`
3. 公开页面路由（示例）：
   - 查询卡片：`/cards/page`
   - 换卡（线上）：`/ONLINE_EYEBALL`、`/ONLINE_EYEBALL/{cardId}`
   - 换卡（线下）：`/EYEBALL`、`/EYEBALL/{cardId}`
   - 签收：`/receipt-public`、`/receipt-public/{cardId}`
4. 线上换卡与线下换卡公开页面路径保持不变，但服务端 endpoint 与页面渲染服务分离实现，避免两个换卡场景的页面字段、脚本和提交逻辑互相耦合。
5. 公开提交与查询路由：
   - `GET /qso-public/records`
   - `GET /overview-public/summary`
   - `GET /exchange-online/-/bureaus`
   - `GET /exchange-online/-/station-cards`
   - `GET /exchange-offline/-/activities`
   - `POST /exchange-online/-/requests`：同呼号存在待审核线上换卡申请时返回 `409/QSL-409-0001`；写入成功后返回本站通信地址。
   - `POST /exchange-offline/-/confirm`
   - `POST /receipt-public/-/confirm`
6. 控制台动作路由：
   - `GET /overview/summary`
   - `GET /reports/summary`
   - `POST /mail-send-confirms/{cardRecordName}/confirm`
   - `POST /mail-receive-confirms/confirm`
   - `POST /mail-receive-confirms/{cardRecordName}/received-date`
   - `POST /exchange-requests/{name}/approve`
   - `POST /exchange-requests/{name}/reject`
   - `POST /exchange-requests/{name}/notify`：发送线上换卡审核通知，成功后写入 `ExchangeRequest.status.reviewMailStatus=SENT` 并阻止重复发送。
   - `POST /notification-mails/send`
   - `POST /notification-mails/batch-send`
   - `POST /notification-mails/test`
   - `POST /imports/precheck`
   - `POST /imports/jobs`
   - `GET /imports/jobs/{jobName}`
   - `GET /imports/jobs/{jobName}/errors`
   - `GET /imports/jobs/{jobName}/errors/download`
   - `POST /exports/jobs`
   - `GET /exports/jobs/{jobName}`
   - `GET /exports/jobs/{jobName}/download`

说明：
1. 卡片路径模式下采用二次确认参数（如 `cs`）进行回填校验；不匹配时页面回填为空白。
2. 公开页面与公开提交 API 分离，页面用于展示与表单，提交落到 `/-/` 子路径接口。
3. 通联业务创建卡片的关联记录为必填；已正式创建卡片或已标记“不创建卡片”的记录不再进入候选列表。“不创建卡片”使用占位 `CardRecord` 持久化，不占用正式卡片编号；占位记录可编辑转为正式卡片，也可删除后释放关联记录候选。
4. 制卡签发、发信确认、送达确认只处理正式卡片编号为 `C{序号}` 的 `CardRecord`，不显示无编号占位记录；控制台发信和送达匹配也排除无编号占位记录。
5. 线下换卡业务不需要打包、制卡邮件流程和收件地址展示；线下换卡制卡签发页面只提供确认制卡，已制卡记录不再因未打包或未发送制卡邮件留在待制卡列表。
6. 线下换卡公开页面 HTML 不直接携带本站联系信息；只有 `POST /exchange-offline/-/confirm` 成功写入后才在响应中返回本站通信地址。
7. 前台线下换卡确认提交成功后同步写入 `CardRecord.spec.cardSent=true`，并在原发卡时间为空时补齐 `sentAt`，使线下换卡业务送达确认中的发卡状态显示为“是”；提交备注写入 `CardRecord.spec.publicReceiptRemarks`，后台线下换卡制卡签发、创建卡片历史详情、送达确认清单等备注查看位置必须展示该字段。线下换卡收卡基本功能必须选择收卡归属活动，该选择同时联动过滤清单；服务端按呼号、卡片类型、场景和关联活动共同匹配待收卡记录。自动创建的临时收卡卡片必须记录关联活动；当前台线下换卡确认激活旧卡片编号时，仅在呼号与关联活动同时匹配时，把临时卡片上的收卡编号原样迁移到旧卡片，并删除临时卡片。
8. 线上换卡公开提交成功后返回本站通信地址用于寄送提示；同一呼号在上一条 `ONLINE_EYEBALL` 申请审核通过或拒绝前不能重复提交。若系统参数 `requiresExchangeReview=false`，提交成功后由系统自动审批通过，审核说明固定为“系统自动审批通过”；人工审核同意时仅在已有说明为空且本次未填写时保持空说明。
9. 线上换卡公开卡片版本接口使用短缓存；Extension 查询短暂失败时，优先返回上一次成功读取的卡片版本列表，避免公开表单因一次后端查询抖动直接不可用。接口只返回卡片预览缩略图地址，不返回附件原图或 base64 图片内容。
10. 收卡业务三个子菜单的基本功能“确认收信”和清单内“确认收卡”必须使用页面上方填写的收卡日期；未填写日期时前端弹窗提示且服务端拒绝提交，不再默认使用当前日期；该日期在本次浏览会话内按收卡子菜单记忆，提交后不清空。同一呼号、卡片类型和场景存在多条卡片记录时，收卡确认只匹配未完成收卡的正式卡片记录，并按卡片编号序号从小到大绑定；更早记录完成收卡后才绑定下一条。已收卡片清单支持将指定收卡编号从源卡片迁移到其他正式卡片 ID 的收卡清单中，迁移后按源/目标剩余编号同步维护收卡状态与收卡邮件状态。
11. 状态流转联动：线下换卡收到对方“已签收”时补齐未完成的 `cardSent/sentAt`；线上换卡收到对方“已签收”时补齐未完成的 `cardIssued/cardIssuedAt/envelopePrinted/cardSent/sentAt`；通联业务和线上换卡业务只要 `cardSent=true`，即联动补齐制卡和打包状态。批量编辑将状态机改为否或空时，对应时间、邮件状态、邮件时间和邮件错误字段同步置空。线上换卡审核通过只代表申请通过和我方待发卡，不代表我方已收到对方卡片。
12. 总览看板与统计报表的卡片类统计项统一排除 `sceneType=EYEBALL` 且呼号为空的卡片记录（线下换卡占位卡），避免占位记录影响业务统计口径。

## 9. 权限与安全
1. 权限遵循 `view/edit` 最小权限模型。
2. 服务端鉴权为准，前端仅做展示与导航控制。
3. 公开接口启用独立限流服务（按 IP/场景键）。
4. 审计日志覆盖关键写操作链路。
5. 本地卡片打印工具提供独立权限模板 `qsl-management-card-print-tool`（显示名“卡片打印工具”），通过依赖汇总 `qsl-management-station-profile-view`、`qsl-management-card-record-edit`、`qsl-management-qso-record-view`、`qsl-management-address-bureau-view`、`qsl-management-offline-activity-view`，便于为 Halo 个人令牌一次性授权打印、查询与回写所需权限；其中 `qsl-management-offline-activity-view` 为隐藏只读模板，仅授予 `offline-activities` 的 `get/list` 权限，用于线下换卡制卡队列活动筛选和 QTH 活动地点填充。

## 11. 本地打印工具（tools/CardPrint）
1. 主入口：
   - `python -m cardprint.cli ui online`
2. 当前作业流：
   - 配置登录 -> 卡片打印 -> 制卡确认 -> 打印封面（打包）-> 打包确认
3. 与后台关系：
   - 读取 QSL 后台卡片与地址信息生成打印队列。
   - 打印状态通过人工确认页签回写，避免自动误回写。
4. 在线桥接代码事实：
   - 默认 `base_url` 为 `http://localhost:8090`，但配置可传入其他地址。
   - 卡片/信封基础数据读取 `card-records`，卡片打印会按 `CardRecord.spec.qsoRecordName` 额外读取 `qso-records` 并注入 `qsoInfo`。
   - 信封补全会读取 `station-profiles`、`address-book-entries`、`bureau-entries`。
   - 卡片版本列表通过公开接口 `GET /apis/api.qsl-management.halo.run/v1alpha1/exchange-online/-/station-cards` 读取，本质数据来源仍为 `station-cards`，不从 `card-records` 反推，并按 `sortOrder` 与版本号排序；本台通信地址仍通过受保护的 `station-profiles` Extension API 读取。
   - 线上换卡申请审核通过自动生成卡片时，按申请地址复用或创建地址资源，并写入卡片记录的 `addressEntryName`。
   - 卡片二维码拼接支持 `qrcode.path_mappings` 短路径映射，在线打印配置页可在站点地址下方配置线下换卡、线上换卡、签收确认三类短路径；默认将公开页面长路径映射为 `/EYEBALL`、`/ONLINE_EYEBALL`、`/rp` 后再追加卡片 ID 与 `cs` 参数。
   - 登录并拉取卡片版本时，网络请求在后台线程执行，避免界面短暂未响应；只补齐空白本台通信地址字段，不覆盖已有本台姓名、电话、邮编、地址。若受保护接口返回登录页或 HTML，工具明确提示认证或权限问题，不再静默解析为空数据。
   - 回写采用 Halo Extension 完整 `PUT`，先读取当前对象再合并 `spec`；`cardIssuedAt` 使用 `yyyy-MM-dd HH:mm:ss` 文本格式；回写会同步刷新 `CardRecord.status.flowStatus`，并按后台状态流转规则补齐或清理相关状态字段。
