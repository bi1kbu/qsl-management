# QSL 项目信息结构化清单

更新时间：2026-05-01

代码核验范围：`qsl-plugin`、`tools/CardPrint`

## 1. 项目定位
1. 项目名称：QSL 管理系统（Halo2 插件 + 本地打印工具）。
2. 核心目标：围绕通联、换卡、制卡签发、发信与送达确认、收卡编号管理，提供可持久化的后台业务流与前台自助页面。
3. 技术形态：
   - Halo2 插件后端：Java 21 + Spring WebFlux + Halo Extension API
   - Halo2 插件前端：Vue 3 + TypeScript
   - 本地打印工具：`tools/CardPrint`（Python）

## 2. 仓库结构
1. `qsl-plugin/`：Halo2 插件工程（后端 + 控制台前端）。
2. `tools/CardPrint/`：本地在线打印工具（独立 Python 项目）。
3. `docs/spec/`：结构化项目信息文档。
4. `ProductDefinition.md`：产品定义（约束性文档）。
5. `BackendApiContract.md`：后端 API 合同（输出性文档，需与代码同步）。
6. `agents.md`：协作与约束规则（约束性文档）。

## 3. 后端架构（qsl-plugin/src/main/java）
1. 插件启动与默认资源：
   - `QslManagementPlugin`：确保默认 `SystemSetting`、`StationProfile` 实例存在。
2. Extension 注册：
   - `QslExtensionSchemeRegistrar`：注册并维护名称索引。
3. 业务服务：
   - `QslConsoleActionService`：后台业务操作（确认、状态流转等）。
   - `QslImportExportJobService`：导入导出任务。
   - `QslNotificationMailService`：邮件发送与状态记录。
   - `QslPublicApiService`：前台公开业务处理。
4. Endpoint 分层：
   - `console.api.qsl-management.halo.run`：控制台 API
   - `api.qsl-management.halo.run`：公开 API 与前台页面路由
5. 前台渲染与短码：
   - `QslPublic*PageRenderService`：前台页面 HTML 渲染
   - `Qsl*EmbedContentTransformer`：文章/页面短码转嵌入卡片

## 4. 前端架构（qsl-plugin/ui/src）
1. 入口：
   - `index.ts`：注册 `/qsl` 路由与菜单分组。
2. 菜单配置：
   - `constants/menu-modules.ts`：模块定义、权限节点、依赖关系。
3. 模块承载：
   - `views/MenuLayoutView.vue`：分组布局
   - `views/MenuModuleView.vue`：按模块 key 分发到具体页面组件
4. 通用组件：
   - `QslBatchFieldEditor`、`QslBusinessRecordHeader`、`QslExpandableHistoryTable`、`QslPaginationBar`、`QslQueryToolbar`。
5. 业务页面：
   - `views/modules/*Module.vue`（通联、卡片、签发、确认、审计、数据、配置等）。

## 6. 核心数据模型（Extension）
1. `SystemSetting`：系统参数（限流、审核开关、序号等）。
2. `StationProfile`：本台通信地址与联系信息。
3. `StationEquipment`：本台设备信息。
4. `StationCard`：本台卡片版本、库存与排序。
5. `QsoRecord`：通联日志。
6. `CardRecord`：卡片主记录（场景、版本、状态、备注、邮件状态、地址绑定、收卡编号）。
7. `ExchangeRequest`：线上换卡申请。
8. `OfflineActivity`：线下换卡活动。
9. `AddressBookEntry`：通信地址簿。
10. `BureauEntry`：卡片局地址簿。
11. `EquipmentCatalogEntry`：设备库。
12. `QslAuditLog`：审计日志。
13. `ImportExportJob`：导入导出任务。

关键新增字段事实：

1. `SystemSetting.spec` 包含 `autoNotifyOnCardCreated/autoNotifyOnCardSent/autoNotifyOnCardReceived/cardRecordSequence/receiveRecordSequence`。
2. `StationCard.spec` 包含库存与排序字段：`availableInventory/versionTotal/sortOrder`。
3. `QsoRecord.spec` 包含 `sceneType/myQth/operator`。
4. `CardRecord.spec` 包含 `sceneType/offlineActivityName/addressEntryName/businessRemarks/createdRemarks/sentRemarks/receivedRemarks/publicReceiptRemarks/cardIssued/envelopePrinted/receiptConfirmed/receivedRecordCodes` 以及三类邮件状态字段。
5. `ExchangeRequest.spec` 包含 `sceneType/cardVersion`。

## 7. 场景主字段约定
1. `sceneType` 使用：`QSO`、`SWL`、`ONLINE_EYEBALL`、`EYEBALL`。
2. 通联业务：主场景 `QSO/SWL`。
3. 线上换卡业务：主场景 `ONLINE_EYEBALL`。
4. 线下换卡业务：主场景 `EYEBALL`。

## 8. 接口边界（按代码当前状态）
1. 控制台 API 前缀：
   - `/apis/console.api.qsl-management.halo.run/v1alpha1/...`
2. 公开 API 前缀：
   - `/apis/api.qsl-management.halo.run/v1alpha1/...`
3. 公开页面路由（示例）：
   - 查询卡片：`/cards/page`
- 换卡（线上）：`/ONLINE_EYEBALL`、`/ONLINE_EYEBALL/{cardId}`
- 换卡（线下）：`/EYEBALL`、`/EYEBALL/{cardId}`
   - 签收：`/receipt-public`、`/receipt-public/{cardId}`
4. 公开提交与查询路由：
   - `GET /qso-public/records`
   - `GET /overview-public/summary`
   - `GET /exchange-online/-/bureaus`
   - `GET /exchange-online/-/station-cards`
   - `GET /exchange-offline/-/activities`
   - `POST /exchange-online/-/requests`
   - `POST /exchange-offline/-/confirm`
   - `POST /receipt-public/-/confirm`
5. 控制台动作路由：
   - `GET /overview/summary`
   - `GET /reports/summary`
   - `POST /mail-send-confirms/{cardRecordName}/confirm`
   - `POST /mail-receive-confirms/confirm`
   - `POST /exchange-requests/{name}/approve`
   - `POST /exchange-requests/{name}/reject`
   - `POST /exchange-requests/{name}/notify`
   - `POST /notification-mails/send`
   - `POST /notification-mails/batch-send`
   - `POST /imports/precheck`
   - `POST /imports/jobs`
   - `GET /imports/jobs/{jobName}`
   - `GET /imports/jobs/{jobName}/errors`
   - `GET /imports/jobs/{jobName}/errors/download`
   - `POST /exports/jobs`
   - `GET /exports/jobs/{jobName}`
   - `GET /exports/jobs/{jobName}/download`

说明：
1. 卡片路径模式下采用二次确认参数（如 `cs`）进行回填校验；不匹配时页面回填为空白。
2. 公开页面与公开提交 API 分离，页面用于展示与表单，提交落到 `/-/` 子路径接口。
3. 通联业务创建卡片的关联记录为必填；已正式创建卡片或已标记“不创建卡片”的记录不再进入候选列表。“不创建卡片”使用占位 `CardRecord` 持久化，不占用正式卡片编号；占位记录可编辑转为正式卡片，也可删除后释放关联记录候选。
4. 制卡签发、发信确认、送达确认只处理正式卡片编号为 `C{序号}` 的 `CardRecord`，不显示无编号占位记录；控制台发信和送达匹配也排除无编号占位记录。

## 9. 权限与安全
1. 权限遵循 `view/edit` 最小权限模型。
2. 服务端鉴权为准，前端仅做展示与导航控制。
3. 公开接口启用独立限流服务（按 IP/场景键）。
4. 审计日志覆盖关键写操作链路。

## 11. 本地打印工具（tools/CardPrint）
1. 主入口：
   - `python -m cardprint.cli ui online`
2. 当前作业流：
   - 配置登录 -> 卡片打印 -> 制卡确认 -> 打印封面（打包）-> 打包确认
3. 与后台关系：
   - 读取 QSL 后台卡片与地址信息生成打印队列。
   - 打印状态通过人工确认页签回写，避免自动误回写。
4. 在线桥接代码事实：
   - 默认 `base_url` 为 `http://localhost:8090`，但配置可传入其他地址。
   - 卡片/信封基础数据读取 `card-records`。
   - 信封补全会读取 `station-profiles`、`address-book-entries`、`bureau-entries`。
- 卡片版本列表读取 `station-cards`，不从 `card-records` 反推。
- 线上换卡申请审核通过自动生成卡片时，按申请地址复用或创建地址资源，并写入卡片记录的 `addressEntryName`。
   - 回写采用 Halo Extension 完整 `PUT`，先读取当前对象再合并 `spec`。
