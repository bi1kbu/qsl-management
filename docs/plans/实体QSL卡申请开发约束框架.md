1. 本文档是“申请实体通联卡片”功能的约束性开发文档，覆盖需求分析、设计、编码、测试、文档同步和验收。
2. 在本文档约束未完整落地前，不得把该功能标记为完成。
3. `agents.md` 与 `docs/spec/ProductDefinition.md` 的全局约束继续生效；若实现过程中发现冲突或需要超出本文档范围，必须先形成偏差表并取得确认，不得以代码事实反向覆盖约束。
4. 本功能不复用 `ExchangeRequest（线上换卡申请）`，必须使用独立数据模型、服务、公开页面和后台审核流程。
5. 本文档只冻结本次功能边界，不授权顺带重构线上换卡、线下换卡、公开签收或本地打印工具的无关代码。

## 3. 功能目标

为用户提供公开的实体 QSL 卡申请页面，使用户能够：

1. 输入 `callSign（呼号）` 查询本站保存的对应 QSO 记录。
2. 勾选一条或多条可申请 QSO。
3. 为每条已勾选 QSO 单独选择一个 `cardVersion（卡片版本）`。
4. 重新填写个人收件地址，或选择公开的现有卡片局。
5. 填写 `notificationEmail（通知电子邮箱）` 和 `remarks（备注）` 后提交申请。
6. 后台审核人员点击“审核通过”后，系统立即为全部申请项目自动创建卡片。
7. 申请记录持久化保存每条 QSO 与自动创建的卡片编号之间的一对一映射。

## 4. 明确不在本次范围内的能力

1. 不创建 `/apis/.../QSL_CARD` 等资源型页面入口。
2. 不创建 `/qc` 或其他极简别名。
3. 不提供前台个人地址查询、地址回显、地址选择或地址自动补全接口。
4. 不允许匿名用户直接创建、编辑或删除 `AddressBookEntry（地址簿条目）`、`BureauEntry（卡片局条目）`、`CardRecord（卡片记录）` 或实体卡申请 Extension。
5. 不允许前台直接匿名新增卡片局。
6. 不复用或扩展 `ExchangeRequest（线上换卡申请）` 承载本功能。
7. 本期仅支持 `sceneType（场景类型）=QSO` 的通联记录，不包含 SWL、线上换卡和线下换卡记录。
8. 不新增卡片局业务编号，不改变现有卡片局资源命名与卡片地址绑定规则。

## 5. 公开页面与路由约束

### 5.1 页面入口

唯一面向用户的页面入口为：

```text
GET /qsl_card
```

约束如下：

1. `/qsl_card` 属于匿名只读非资源型 API。
2. 仅允许精确的 `GET` 请求，可兼容末尾单个 `/`，不得接受任意子路径。
3. 浏览器地址不得跳转到 `/apis/...`。
4. 页面不得依赖同功能的 `/apis/...` HTML 页面 Endpoint。
5. 推荐新增独立 `AdditionalWebFilter`，在 Halo 授权完成后由专用页面渲染服务输出 HTML。
6. 页面渲染服务、公开数据 Endpoint 和业务服务必须分离，不得把 HTML、数据查询和提交写入同一个大类。
7. 面向用户的整个页面必须使用中英双语，不展示 `addressType`、`postalCode` 等内部字段名；标题、说明、分区标题、表头、字段标签、按钮、状态、校验提示、加载提示、错误提示和提交结果均采用中文在上、英文在下的两行布局。原生下拉选项受浏览器控件限制时，允许在同一选项内使用“中文 / English”，但对应字段标签和辅助说明仍必须使用上下两行。

### 5.2 Halo 扩展点偏差记录

使用 `AdditionalWebFilter` 直接承载完整页面并非 Halo 文档中最典型的页面注册方式，但本功能明确禁止创建 `/apis` 页面入口，因此允许采用该方案。

| 偏差项 | 原因 | 风险 | 控制措施 | 回退方案 |
| --- | --- | --- | --- | --- |
| `AdditionalWebFilter` 直接输出 `/qsl_card` 页面 | 用户页面只能使用非资源型短路径 | 过滤器顺序错误、路径冲突、绕过授权 | 精确匹配 GET、授权后执行、专项路由与 RBAC 测试 | 若 Halo 后续提供官方根路径页面扩展点，迁移到官方扩展点并保持 `/qsl_card` 不变 |

## 6. 前台交互流程

### 6.1 查询 QSO

1. 用户填写 `callSign（呼号）`，前端统一去除首尾空格并转为大写。
2. 用户主动点击“查询通联记录”，不得在输入过程中自动高频查询。
3. 查询结果必须显示全部匹配 QSO，包括不可申请的记录。
4. 每条记录至少展示：
   - `qsoRecordName（QSO记录编号）`
   - `date（通联日期）`
   - `time（通联时间）`
   - `freq（通联频率）`
   - `qth（对方位置）`
   - `selectable（是否可选择）`
   - `unselectableReason（不可选择原因）`
   - `statusCode（公开状态码）`
   - `statusText（中文状态文字）`
   - `statusTextEn（英文状态文字）`
5. 公开结果不得包含本台设备、完整备注、个人地址、联系方式、邮件状态或其他无关敏感字段。

### 6.2 QSO 选择与卡片版本

1. 可申请 QSO 使用复选框选择。
2. 已勾选的每条 QSO 必须单独选择一个有效 `cardVersion（卡片版本）`。
3. 未勾选记录不要求选择卡片版本。
4. 取消勾选时必须清除该行已选择的卡片版本，提交载荷不得包含未勾选记录。
5. 同一个卡片版本可以用于多条 QSO，但服务端必须按本次申请的合计数量检查库存。
6. 前端库存提示只用于交互，最终有效性和库存必须由服务端重新校验。
7. `StationCard.status.active=true` 表示“仅支持实体 QSL 通联卡”，不限制本页面选择和提交；`false/null` 表示通用版本。该字段只限制线上换卡，普通通联和线下换卡与实体 QSL 卡申请保持一致，均允许使用。
8. 勾选可申请 QSO 后，页面必须在该记录下方展开可视化卡片版本候选区，不使用仅显示文字的原生下拉框。每个候选项左侧使用固定正方形图片框，以 `object-fit: contain` 完整显示横版或竖版卡面；右侧显示卡片版本名称和剩余可用数量。每条 QSO 使用独立单选状态，取消勾选时收起候选区并清除已选版本。
9. QSO 表格最右侧必须提供“申请此记录的卡片”按钮。可申请记录点击该按钮时只激活同一行左侧复选框并展开卡片版本候选区，不直接提交申请；勾选后按钮显示“已选择此记录”并禁用，取消复选框后恢复。审核中、建卡中、建卡异常、待制卡、已制卡、已打包、已发卡、已签收或已有卡片的记录保留按钮位置但必须禁用；申请未通过且没有卡片的记录允许重新申请。

### 6.3 前台申请与发卡进度状态

公开查询必须根据实体卡申请和关联卡片的实时事实计算状态，不得把展示状态重复写入 QSO 或申请记录：

| `statusCode（状态码）` | 页面中文状态 | 页面英文状态 | 是否可选择 | 服务端判断依据 |
| --- | --- | --- | --- | --- |
| `AVAILABLE` | 可申请 | Available | 是 | 无有效占用、申请或关联卡片 |
| `REJECTED_REAPPLY` | 申请未通过，可重新申请 | Not Approved, Reapplication Available | 是 | 最近申请已拒绝且没有关联卡片或有效占用 |
| `REVIEW_PENDING` | 审核中 | Pending Review | 否 | 存在有效占用，或申请审核状态为待处理 |
| `CARD_CREATING` | 审核通过，正在创建卡片 | Approved, Creating Card | 否 | 审核通过且建卡状态为创建中 |
| `CARD_CREATION_FAILED` | 审核通过，建卡异常 | Approved, Card Creation Issue | 否 | 审核通过且该申请存在建卡失败 |
| `CARD_PENDING_ISSUE` | 审核通过，待制卡 | Approved, Awaiting Card Production | 否 | 审核通过且卡片已创建、尚未制卡 |
| `CARD_ISSUED` | 已制卡，待打包 | Card Produced, Awaiting Packing | 否 | `CardRecord.spec.cardIssued=true` |
| `CARD_PACKED` | 已打包，待寄出 | Packed, Awaiting Dispatch | 否 | `CardRecord.spec.envelopePrinted=true` |
| `CARD_SENT` | 已发卡 | Card Sent | 否 | `CardRecord.spec.cardSent=true` |
| `CARD_SIGNED` | 已签收 | Delivered | 否 | `CardRecord.spec.receiptConfirmed=true` |

补充约束：

1. 后台直接创建或旧数据无法关联到实体卡申请时，尚未制卡的卡片显示“已有卡片，待制卡 / Card Exists, Awaiting Production”，状态码仍为 `CARD_PENDING_ISSUE`，不得错误宣称已经通过申请审核。
2. “已有卡片”只作为无法进一步判断的旧数据兜底，不再作为正常业务流程的唯一状态。
3. “已收卡片”表示本站收到对方卡片，不属于用户申请实体卡的寄送进度，不进入本状态链。
4. 同一 QSO 存在多张历史卡片时，按“已签收→已发卡→已打包→已制卡→待制卡”的最高完成度展示。
5. `unselectableReason（不可选择原因）` 为兼容字段；不可选择时使用中文状态文字，可重新申请或可申请时保持为空。

### 6.4 地址类型

页面提供以下两种地址类型：

1. `PERSONAL（个人地址）`
2. `BUREAU（卡片局）`

切换地址类型时，前端必须清理另一类型的提交字段，避免同时提交个人地址和卡片局。

### 6.5 个人地址流程

1. 前台不得从后台获取任何个人地址。
2. 每次申请必须重新填写个人地址，不提供历史地址下拉框。
3. 必填字段为：
   - `postalCode（邮政编码）`
   - `address（通信地址）`
   - `notificationEmail（通知电子邮箱）`
4. 选填字段为：
   - `name（收件人姓名）`
   - `telephone（联系电话）`
5. 提交后由服务端按标准化字段精确查重，复用或创建 `AddressBookEntry（地址簿条目）`；选填字段为空时按空值参与精确匹配。
6. 地址资源保存姓名、电话、邮编、地址和电子邮箱等联系方式。
7. 申请记录不复制姓名、电话、邮编和地址，只保存 `addressEntryName（地址内部资源名称）`。
8. `notificationEmail（通知电子邮箱）` 必须在申请记录中额外保留一份，作为本次申请生命周期内固定的邮件通知目标。
9. 页面和提交响应不得返回地址内部资源名称，也不得提示后台是否复用了已有地址。
10. 无论最终审核通过或拒绝，提交时已创建或复用的地址资源均保留，不因申请拒绝而自动删除。

### 6.6 现有卡片局流程

1. 卡片局清单属于公开数据，可以从现有公开 API 获取。
2. 页面显示去向国、卡片局名称、邮编和地址，选项顺序固定为“去向国｜卡片局名称｜邮编｜地址”；去向国未配置时显示“未设置去向国 / Country Not Set”。不展示电话、备注、同步状态等后台字段。
3. 前端提交现有 `BureauEntry.metadata.name（卡片局内部资源名称）`，不得以页面提交的名称、邮编和地址直接覆盖后台卡片局数据。
4. 服务端必须根据内部资源名称重新读取并验证卡片局存在。
5. 不新增卡片局业务编号字段；内部仍沿用现有 `BureauEntry.metadata.name`。
6. 卡片和申请继续沿用现有地址绑定方式，将卡片局内部资源名称写入 `addressEntryName（地址绑定）`。

### 6.7 新增卡片局联系流程

1. 卡片局选择区域提供“申请新增卡片局”入口。
2. 选择该入口后，不得直接提交实体卡申请，也不得匿名创建 `BureauEntry`。
3. 页面通过公开数据 API 获取 `StationProfile.spec.myEmail（本台电子邮箱）`，只返回该邮箱，不返回本台电话、通信地址等其他资料。
4. 页面提示对方通过邮件联系本台，并在邮件中填写计划使用的卡片局资料。
5. 页面可以提供 `mailto:` 链接，建议预填：
   - 邮件主题：`实体QSL卡申请新增卡片局`
   - 用户呼号
   - 计划使用的卡片局名称
   - 卡片局邮编
   - 卡片局地址
6. 本台在后台按现有卡片局管理流程确认并新增后，用户重新进入页面，从公开卡片局清单中选择后提交。
7. 如果本台电子邮箱未配置，页面必须提示“本台暂未配置联系邮箱”，不得显示空 `mailto:` 链接。

### 6.8 提交成功

提交成功后只向用户返回并展示：

1. `requestName（申请编号）`
2. `callSign（呼号）`
3. `reviewStatus（审核状态）`
4. `submittedAt（提交时间）`
5. 已选择 QSO 的数量

不得在成功响应中返回个人地址、地址内部资源名称、后台审核信息或卡片局内部资源名称。

## 7. 持久化数据模型

### 7.1 实体卡申请

新增独立 Extension：

```text
group: qsl-management.bi1kbu.com
version: v1alpha1
kind: QslCardRequest
plural: qsl-card-requests
singular: qsl-card-request
```

申请资源名称作为申请编号，冻结为 `QCR{四位以上递增序号}`，例如 `QCR0001`。编号由服务端生成，前端不得指定。

建议模型：

```text
QslCardRequest.spec
  callSign（呼号）
  qsoItems（申请项目清单）
    qsoRecordName（QSO记录编号）
    cardVersion（卡片版本）
  addressType（地址类型：PERSONAL/BUREAU）
  addressEntryName（地址或卡片局内部资源名称）
  notificationEmail（通知电子邮箱）
  remarks（申请备注）
  submittedAt（提交时间）

QslCardRequest.status
  reviewStatus（审核状态：待处理/通过/拒绝）
  reviewReason（审核说明）
  reviewedBy（审核人）
  reviewedAt（审核时间）
  cardCreationStatus（卡片创建状态：未创建/创建中/全部成功/部分失败）
  createdCards（卡片创建结果）
    qsoRecordName（QSO记录编号）
    cardVersion（卡片版本）
    cardRecordName（卡片编号）
    creationStatus（创建状态：成功/失败）
    lastError（最近错误）
  reviewMailStatus（审核邮件状态）
  reviewMailSentAt（审核邮件发送时间）
  reviewMailLastError（审核邮件最近错误）
  reviewMailTargetEmail（审核邮件目标邮箱）
```

约束如下：

1. `qsoItems（申请项目清单）` 至少一项，QSO 编号不得重复。
2. 每项必须同时包含 QSO 编号和卡片版本。
3. `addressType（地址类型）` 与 `addressEntryName（地址绑定）` 必须一致且可解析。
4. `notificationEmail（通知电子邮箱）` 为必填，邮件发送始终以申请保存值为准，不因地址簿后续修改而改变。
5. `reviewStatus（审核状态）` 只允许“待处理”“通过”“拒绝”三类。
6. 卡片创建技术状态不得扩展为新的审核状态。

### 7.2 QSO 持久化占用

仅在提交时扫描待处理申请无法防止并发重复提交，因此必须设计持久化 QSO 占用记录。推荐新增独立 Extension：

```text
kind: QslCardRequestQsoReservation
plural: qsl-card-request-qso-reservations
```

至少保存：

```text
requestName（申请编号）
qsoRecordName（QSO记录编号）
createdAt（占用时间）
```

占用规则：

1. 每个 QSO 只能存在一个有效占用记录。
2. 提交申请时先检查已有卡片，再原子创建占用记录。
3. 多条 QSO 中任一占用失败，本次申请不得创建，并补偿删除本次已经创建的占用记录。
4. 申请被拒绝后释放全部占用，使 QSO 可以再次申请。
5. 申请通过且卡片创建完成后，可以删除占用；后续由“已有卡片”规则阻止重复申请。
6. 插件启动或管理员动作必须能够识别孤立占用，并根据申请和卡片事实进行修复，不得永久锁死 QSO。

## 8. 提交服务端校验

提交接口必须在服务端执行以下全部校验：

1. 呼号格式、长度和标准化结果合法。
2. QSO 清单非空、数量在系统允许范围内且没有重复项。
3. 每条 QSO 存在，属于提交呼号，且 `sceneType（场景类型）=QSO`。
4. 每条 QSO 尚未被任何卡片记录占用。
5. 每条 QSO 尚未被其他待处理申请占用。
6. 每条 QSO 都选择一个存在、可用且库存充足的卡片版本。
7. 同一卡片版本被多条 QSO 使用时，按合计申请数量检查库存。
8. 个人地址字段完整且长度、电话、邮编、邮箱格式合法。
9. 卡片局模式下只能提交现有卡片局内部资源名称，且服务端重新读取成功。
10. 备注长度、字符和请求体大小符合限制。
11. 页面上的禁用、必填和库存判断只用于用户体验，不得作为服务端校验替代品。

## 9. 审核与自动创建卡片

### 9.1 后台入口

后台新增：

```text
通联业务 → 实体卡申请审核
```

页面至少支持：

1. 按审核状态筛选。
2. 查看申请编号、呼号、通知邮箱、地址类型、地址绑定、备注和提交时间。
3. 展开查看每条 QSO、所选卡片版本和卡片创建结果。
4. 审核通过。
5. 审核拒绝并填写原因。
6. 对部分失败的卡片创建结果执行幂等重试。

### 9.2 审核通过

点击“审核通过”后必须由同一个服务端业务操作立即尝试创建全部卡片，不再提供单独的首次“创建卡片”按钮。

执行顺序：

1. 锁定申请，阻止重复审核和重复点击。
2. 确认申请当前状态为“待处理”。
3. 重新校验全部 QSO、占用关系、已有卡片、卡片版本和库存。
4. 业务校验失败时不得创建任何新卡片，申请保持“待处理”，向审核人员返回明确原因。
5. 业务校验通过后，把审核状态更新为“通过”，同时进入卡片创建流程。
6. 为每条申请项目创建一张 `CardRecord（卡片记录）`。
7. 将每条创建结果写回 `createdCards（卡片创建结果）`。
8. 全部成功时设置 `cardCreationStatus（卡片创建状态）=全部成功`。
9. 运行时发生部分失败时保留已创建卡片，设置为“部分失败”，允许后台幂等重试失败项。
10. 审核完成后按通知策略向申请保存的电子邮箱发送结果邮件。

### 9.3 自动创建的卡片字段

每条自动创建的卡片至少写入：

```text
callSign（呼号）=<申请呼号>
cardType（卡片类型）=QSO
sceneType（场景类型）=QSO
cardVersion（卡片版本）=<该QSO选择的版本>
qsoRecordName（QSO记录编号）=<对应QSO>
addressEntryName（地址绑定）=<申请地址或卡片局内部资源名称>
mailTargetEmail（邮件目标邮箱）=<申请通知邮箱>
businessRemarks（业务备注）=<包含申请编号和申请备注>
cardRemarks（卡片备注）=<通联业务现行默认卡片备注>
cardReceived（是否已收卡）=false
```

### 9.4 幂等与异常恢复

1. 同一申请、同一 QSO 不得创建第二张卡片。
2. 重试前必须先检查申请状态中的卡片映射和 `CardRecord.spec.qsoRecordName`。
3. 如果卡片已经创建但申请状态尚未成功回写，重试必须识别并补写映射，不得再创建卡片。
4. 卡片业务备注或元数据应保留申请编号，用于崩溃恢复和审计追踪。
5. 幂等重试只处理失败项，不重复处理已经成功的项目。

## 11. 安全设计

### 11.1 认证与鉴权

1. `/qsl_card` 只授予匿名 `GET` 非资源型权限。
2. 公开数据 API 只授予所需的 `get/create` 动作，不授予 Extension 通配权限。
3. 后台查看和编辑权限必须分离，编辑依赖查看。
4. 审核、拒绝、重试和卡片创建必须执行服务端鉴权，前端隐藏按钮不能作为安全边界。

### 11.2 个人信息保护

1. 不存在任何按呼号查询个人地址的公开接口。
2. 公开 QSO 查询只返回申请所需最小字段。
3. 申请成功响应不得回显个人地址。
4. 日志不得打印完整地址、电话或电子邮箱；审计记录需要邮箱时应脱敏。
5. `notificationEmail（通知电子邮箱）` 只用于本次申请的邮件通知和审计，不得扩展为公开身份查询条件。

### 11.3 匿名攻击面

1. QSO 查询、公开邮箱读取和申请提交分别配置限流键。
2. 限流至少按 `endpoint + clientIp`，提交还应结合标准化呼号和 QSO 集合进行短期重复抑制。
3. 限制请求体大小、QSO 数量、字符串长度和非法字符。
4. 服务端输出 HTML 和错误提示时必须转义用户输入，防止 XSS。
5. 公开邮箱属于用户明确要求公开的信息，应记录邮件地址抓取和垃圾邮件风险；页面仅在“申请新增卡片局”流程中展示。
6. 提交来源 IP 不写入申请业务字段，按现有规则进入审计日志或安全日志。

### 11.4 并发安全

1. 进程内锁只能作为降低重复请求的辅助措施，不得作为唯一一致性保障。
2. QSO 持久化占用和卡片存在性校验共同构成重复提交安全边界。
3. 审核通过动作必须按申请加锁并支持幂等重试。

## 12. RBAC 角色模板

新增后台权限节点：

```text
plugin:qsl-management:qsl-card-request:view
plugin:qsl-management:qsl-card-request:edit
```

依赖关系：

1. `qsl-card-request:view` 至少具备申请、关联 QSO、卡片版本、地址/卡片局和已创建卡片的只读权限。
2. `qsl-card-request:edit` 依赖 `qsl-card-request:view`，并具备审核动作、卡片创建和审核邮件发送所需的最小写权限。
3. 不得复用权限含义不相符的 `exchange-request-review` 作为本功能唯一权限节点。
4. 匿名角色只能获得 `/qsl_card` 和本功能公开数据 API 的精确权限。

## 13. 邮件通知

1. `notificationEmail（通知电子邮箱）` 是申请必填字段，并保存在申请记录中。
2. 审核通过或拒绝的邮件目标始终使用申请保存的邮箱快照。
3. 审核通过邮件应包含申请编号、审核结果、成功创建的卡片编号及对应 QSO。
4. 部分创建失败时，邮件内容不得宣称全部卡片已创建成功；是否自动发送失败补充通知应由现有通知策略或新增明确策略控制。
5. 邮件状态和最近错误必须持久化，已经成功发送的同一场景邮件默认不得重复自动发送。
6. “申请新增卡片局”只提供用户主动发送邮件的 `mailto:` 入口，不由插件代替用户自动外发邮件。

## 14. 数据迁移、导入导出与审计

1. 新增 `QslCardRequest` 和 QSO 占用模型时必须提升数据模型版本。
2. 必须通过 `QslMigrationState` 记录：
   - 运行插件版本
   - 数据模型版本
   - 新模型注册与确认迁移项
   - 最近迁移结果
3. 首次安装只建立当前模型基线；存量安装执行幂等模型确认，不通过一次性前端脚本判断迁移状态。
4. 全量导出和恢复必须包含实体卡申请、逐条卡片映射及有效 QSO 占用记录。
5. 导入后必须执行占用、申请和卡片之间的一致性重建或校验。
6. 提交、审核、拒绝、自动建卡、部分失败重试和占用修复必须写入 `QslAuditLog（审计日志）`。
7. 地址和卡片局继续纳入现有导入导出数据集，不为本功能复制新的地址数据集。
