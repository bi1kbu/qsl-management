# QSL 插件下一步开发顺序（一期）

更新时间：2026-04-14

当前基线：后台菜单结构与页面原型已完成；大部分页面仍为前端本地草稿数据，后端实体、持久化、真实 API、服务端鉴权尚未落地。

## 执行约束

1. 每个阶段开始前先核验 Halo2 官方文档（插件开发、扩展点、API 规范、安全建议），并记录核验日期。
2. 先产出设计基线，再编码；未完成基线不进入实现。
3. 权限始终按 `view/edit` 双节点执行，且以服务端鉴权为准。

## 开发顺序

1. **阶段 0：官方资料核验与技术基线定稿**
   - 输出：接口选型、API 分组与版本、认证鉴权方案、权限依赖、风险与回退方案。
   - 完成判定：形成可评审的基线文档并确认。

2. **阶段 1：后端领域模型与存储层**
   - 范围：系统参数、通信地址、本台设备、本台卡片、设备库、地址簿、卡片局、QSO、卡片记录、换卡申请、审计日志。
   - 输出：实体/资源模型、仓储层、基础校验与索引策略。
   - 完成判定：可通过 API 读写核心对象并完成基础约束校验。

3. **阶段 2：后端 API 合同与服务端权限闸门**
   - 范围：按菜单模块提供 `/apis/<group>/<version>/...` 接口，补齐查询、分页、筛选、创建、更新。
   - 输出：OpenAPI/接口清单、统一错误码、`view/edit` 服务端鉴权。
   - 完成判定：所有一期菜单均有对应 API，且越权请求被服务端拒绝。

4. **阶段 3：配置模块联调（配置一级菜单）**
   - 范围：系统参数、通信地址、本台设备、本台卡片。
   - 输出：前后端联调完成，保存/读取走真实 API。
   - 完成判定：配置变更可持久化，页面刷新后数据一致。

5. **阶段 4：数据模块联调（数据一级菜单）**
   - 范围：地址管理、卡片局管理、设备库维护、导入导出（先打通基础上传下载接口）。
   - 输出：主数据 CRUD、分页查询、批量导入导出任务骨架。
   - 完成判定：主数据可被业务模块可靠引用。

6. **阶段 5：业务模块联调（业务一级菜单）**
   - 范围：通联记录、卡片记录、发信确认、收信确认。
   - 输出：业务规则落地（自动补全、关联关系、状态流转）。
   - 完成判定：从录入到发收确认形成可执行闭环。

7. **阶段 6：审核与审计模块联调**
   - 范围：换卡申请审核、通联记录查询、卡片记录查询、统计报表、审计日志。
   - 输出：审批流、查询筛选、统计聚合、不可删除审计留痕。
   - 完成判定：关键操作均产生审计记录，统计与业务数据一致。

8. **阶段 7：导入导出正式化**
   - 范围：单项 CSV 导出、全量 ZIP 导出、CSV 类型自动识别、ZIP 选择性导入。
   - 输出：真实数据读写、预检报告、错误行回执、任务状态跟踪。
   - 完成判定：可稳定处理真实数据，不再依赖前端模拟数据。

9. **阶段 8：前台开放能力（一期最小可用）**
   - 范围：通联查询、换卡申请、卡片签收、数据总览接口。
   - 输出：匿名访问策略、限流、输入校验、防滥用。
   - 完成判定：前台功能可用且满足安全约束。

10. **阶段 9：测试、验收与发布准备**
   - 范围：单元测试、集成测试、权限回归、导入导出压力测试、构建发布。
   - 输出：测试报告、发布说明、升级与回滚说明。
   - 完成判定：`./gradlew build` 通过，核心回归用例通过，可打包交付。

## 里程碑建议

- 里程碑 B：阶段 3-5 完成（后台业务闭环可用）
- 里程碑 C：阶段 6-7 完成（审计与数据交换可用）
- 里程碑 D：阶段 8-9 完成（一期可发布）
