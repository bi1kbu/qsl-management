# QSL 管理系统插件需求说明 v1

## 1. 项目目标

基于 Halo 2 插件框架开发一个 QSL 卡片管理系统，用于管理业余无线电通联卡片（QSL 卡）的录入、查询、寄送、签收、换卡申请、导入导出、统计展示与审计日志。

开发目标：
- 符合 Halo 2 常规 Java 插件开发方式。
- 先在本地开发和验证，确认无误后再部署上线。
- 采用重构重头开发方式，不继承旧系统实现。
- 当前阶段优先完成一期核心功能，外部依赖与智能识别能力作为辅助手段，后续迭代完善。

## 2. 技术与开发约束

- 技术路线：采用 Halo 2 常规 Java 插件方案开发。
- 运行方式：优先使用本地开发模式（development 模式）。
- 开发环境：优先使用 Docker Desktop 搭建 Halo 本地开发环境。
- 插件结构、开发方式、清单配置等需符合 Halo 2 官方要求。
- 当前阶段所有读取和写入文件默认使用 UTF-8 编码。

参考文档：
- Halo 核心框架：https://docs.halo.run/developer-guide/core/framework/
- 插件结构：https://docs.halo.run/developer-guide/plugin/basics/structure/
- 插件清单与 development 模式：https://docs.halo.run/developer-guide/plugin/basics/manifest/
- 开发准备：https://docs.halo.run/developer-guide/plugin/prepare/
- Hello World：https://docs.halo.run/developer-guide/plugin/hello-world/

## 4. 二期范围

二期或后续迭代内容：
- QRZ.cn、QRZ.com 等外部平台自动获取地址信息
- 使用大模型对原始地址文本进行辅助解析
- 更复杂的自动识别与批量填充能力
- 其他外部依赖集成

说明：
- 地址自动获取与大模型解析仅作为辅助手段。
- 手工录入始终为第一优先级。
- 自动解析后必须保留原始文本，供人工二次核对。
- 二期功能在一期中不预留入口、不预留占位页面、不预留占位配置。

## 5. 用户与权限模型

系统角色分为四类：
- 游客：可使用前台公开查询功能和公开统计报表查看功能。
- HAM：基于 Halo 原生注册用户，且已通过呼号确认。HAM 用户可发起换卡申请、补卡申请、查看自己呼号的详细记录并查看自己的申请记录。
- 操作员：可使用后台业务功能，如录入、导出、发信确认、收信确认等。
- 管理员：拥有全部后台功能，包括系统配置、审核、日志查看、权限管理相关能力。

补充规则：
- Halo 原生注册用户若未通过呼号确认，则权限等同于游客。
- 一个用户可绑定多个呼号。
- 每个呼号都必须单独提交证明并分别审核。
- 呼号真实性确认方式：
  - 上传证明截图，例如 hamcq 已验证截图或个人电台执照照片。
  - 或输入后台已记录过的手机号进行辅助核验。
- 手机号辅助核验时，即使匹配到多个呼号或多个地址，也直接视为具备授权并自动通过。
- 呼号绑定审核与角色授权通过 Halo 用户管理功能实现。
- 插件侧只需提供权限节点，数量不限，由管理员在 Halo 用户组中分配权限。
- 权限节点粒度精确到子菜单一级，按功能场景划分，无需细化到按钮级或字段级。

## 7. 状态机设计

### 7.1 卡片状态
建议拆分为三个独立状态域：

1. 制作状态
- 草稿
- 待打印
- 已打印

2. 寄出状态
- 未寄出
- 已寄出

3. 收卡状态
- 未收到
- 已收到

补充字段建议：
- 打印时间
- 打印人
- 寄出时间
- 寄出确认人
- 寄出批次号
- 收到时间
- 收到确认人
- 收卡备注

### 7.2 换卡申请状态
- 待审核
- 已通过
- 已拒绝
- 已完成

补充规则：
- 后台可配置换卡申请是否需要审核。
- 若配置为自动通过，则申请提交后直接进入“已通过”。
- 普通换卡申请通过后自动生成眼球卡记录。
- 补卡申请通过后不新建卡片记录，仅将关联卡片记录的卡片状态与信封状态调整为待打印，寄出状态视同未发/待发。
- 补卡后清空原打印时间、原打印人、原寄出时间、原寄出批次号等当前状态字段，但保留操作日志。
- 若被拒绝，前台需显示拒绝理由。

### 7.3 导入导出任务状态
- 待执行
- 执行中
- 已完成
- 已失败

### 7.4 呼号绑定审核状态
- 待审核
- 已通过
- 已拒绝

## 8. 去重规则

### 8.1 通联记录去重
去重键：
- 对方呼号 + 日期 + 时间 + 频率 + 模式

### 8.2 卡片记录去重
去重键：
- 卡片类型 + 对方呼号 + 日期 + 时间 + 关联通联 ID

### 8.3 地址簿去重
去重键：
- 呼号 + 地址

### 8.4 导入覆盖规则
- 新数据覆盖旧数据。
- 空值不覆盖非空值。
- 人工修改的数据允许被导入覆盖。

## 9. 后台功能需求

### 9.1 本站配置
管理内容：
- 本台呼号
- 姓名
- 电话
- 邮编
- 收件地址
- 备注描述
- 本台设备列表
- 本台天线列表
- 本台功率列表

### 9.2 其他配置
管理内容：
- 卡片局配置
- 前台查询频次限制
- 换卡申请审核策略
- 是否启用补卡
- 补卡间隔天数

前台查询频次限制规则：
- 按分钟限制查询次数。
- 默认值为每分钟 5 次。
- 游客按 IP 限流。
- 登录用户按用户 ID 限流。
- 需支持后台调整。

补卡规则：
- 补卡功能可由后台启用或关闭。
- 一般发卡超过 7 天未收到时视为可能丢件，可申请补卡。
- 补卡间隔天数默认值为 7 天，需支持后台配置。

### 9.3 总览
功能要求：
- 统一展示通联记录、卡片记录、申请记录等核心数据。
- 支持按呼号、卡片类型、制作状态、寄出状态、收卡状态、日期范围等维度筛选。
- 一期优先采用固定字段列表 + 展开详情方式。
- 支持按当前筛选结果导出列表，导出格式为 CSV。
- 总览导出属于数据列表导出，与图表报表导出无关。

### 9.4 通联录入
字段要求：
- 日期
- 时间，默认自动填充但可手动修改
- 时区（当前先支持 UTC / UTC+8）
- 频率
- 模式
- 电台位置
- 本台设备（下拉）
- 本台天线（下拉）
- 对方呼号
- 本台给对方的信号报告
- 对方给本台的信号报告
- 本台功率（下拉）
- 对方功率
- 备注

默认值规则：
- CW 模式下，信号报告默认 599。
- 其他模式默认 59。

### 9.5 卡片录入
字段要求：
- 卡片类型（通联卡 / 收听卡 / 眼球卡）
- 日期
- 时间
- 时区
- 对方呼号
- 姓名
- 电话
- 邮编
- 地址
- 地址原文
- 地址来源
- 备注

业务规则：
- 通联卡、收听卡使用关联通联记录的日期时间。
- 眼球卡默认自动填充当前日期时间，但可手动修改。
- 姓名、电话、邮编、地址支持手工填写。
- 可选接入 QRZ.cn、QRZ.com 获取原始信息。
- 可选调用模型对原始地址文本进行解析。
- 自动获取和自动解析结果必须允许人工修改。

### 9.6 卡片导出
用途：
- 为用户已有 Word 卡片模板提供 CSV 数据源，用于后续手工邮件合并批量打印。

导出方式：
- 按呼号或按记录多选。
- 导出格式为 CSV。
- 编码固定为 UTF-8 with BOM。
- 使用一套卡片模板字段。

导出字段：
- 呼号
- 日期
- 时间
- 时区
- 卡片类型（通联卡 / 收听卡 / 眼球卡）
- 频率
- 模式
- 电台位置
- 本台设备
- 本台天线
- 对方呼号
- 本台给对方的信号报告
- 对方给本台的信号报告
- 本台功率
- 对方功率
- 是否已经收到卡片
- 备注

操作要求：
- 支持导出后标记为已打印。
- 记录打印时间与打印人。

### 9.7 信封导出
用途：
- 为用户已有 Word 信封模板提供 CSV 数据源，用于后续手工邮件合并批量打印。

导出方式：
- 按呼号或按记录多选。
- 导出格式为 CSV。
- 编码固定为 UTF-8 with BOM。
- 使用一套信封模板字段。

导出字段：
- 呼号
- 姓名
- 电话
- 邮编
- 收件地址

操作要求：
- 支持导出后标记为已打印。
- 记录打印时间与打印人。

### 9.8 发信确认
功能要求：
- 通过呼号或记录选择卡片条目。
- 发信确认后自动补全“已打印卡片”和“已打印信封”状态。
- 不允许例外。
- 记录发信确认时间。
- 记录发信确认人。
- 记录寄出批次号。
- 若为补卡发信，则对应卡片记录的补卡次数在实际再次发信成功时加 1。
- 发信结果需供前台公开查询使用。

### 9.9 收信确认
功能要求：
- 通过呼号或记录选择卡片条目。
- 支持对既有卡片记录执行收信确认。
- 支持创建新的待发眼球卡条目，或跳转到卡片录入创建。
- 不允许自动补建通联卡或收听卡。
- 若缺少通联卡或收听卡对应记录，必须先手工创建后，再重新执行收信确认流程。
- 收信确认后自动标记为已收到卡片。
- 记录收到时间。
- 记录收信确认人。
- 支持填写收卡备注。

### 9.10 导入导出
导出要求：
- 支持完整数据备份导出。
- 导出文件需适合后续恢复或迁移。

导入要求：
- 支持从备份文件导入。
- 支持去重更新。
- 按“新数据覆盖旧数据、空值不覆盖非空值、人工修改可被覆盖”的规则处理冲突。
- 支持导入结果摘要与错误报告。
- 支持记录导入任务日志。

### 9.11 删除管理
功能要求：
- 通联记录与卡片记录采用逻辑删除。
- 删除操作必须写入审计日志。
- 删除后默认不在普通列表中显示，但管理员可按条件查看已删除记录。

### 9.12 统计报表
功能要求：
- 提供游客可查看的公开统计报表。
- 提供后台可查看的完整统计报表。
- 前台显示全部公开统计项，不限制时间范围。
- 后台支持按筛选条件查看统计结果，不预设默认时间范围或默认分组维度。
- 报表仅在线查看，不提供导出功能。
- 报表形式为图表和数字数据，包含饼图等统计图形。

建议一期统计项：
- 已发出卡片数量
- 已收到卡片数量
- 待打印数量
- 待寄出数量
- 待收卡数量
- 按月份统计的发卡数量
- 按月份统计的收卡数量
- 按卡片类型统计数量

## 10. 前台功能需求

### 10.1 游客呼号查询
功能要求：
- 游客可通过呼号查询相关卡片记录。
- 查询范围包含眼球卡、通联卡、收听卡。
- 列出查询关键词（呼号）相关的每条记录。

展示信息：
- 呼号
- 卡片类型
- 是否已寄出
- 寄出时间
- 是否已收到
- 收到时间
- 必要通联时间

隐私要求：
- 不显示地址、电话、邮编、姓名等个人信息。

安全要求：
- 需增加频次限制。
- 默认每分钟最多 5 次。
- 游客按 IP 限流。
- 限制值可在后台配置。

### 10.2 HAM 用户详细查询
功能要求：
- HAM 用户可查询自己已绑定呼号的详细记录。
- 详细记录范围包括 QsoRecord 与 QslCardRecord。
- 仅允许查看与本人已审核通过呼号相关的数据。

### 10.3 HAM 用户换卡申请
功能要求：
- 仅 HAM 用户可发起换卡申请。
- 支持普通换卡申请。
- 支持补卡申请。
- 普通换卡申请仅用于眼球卡。
- 补卡申请仅允许呼号持有人（HAM）或操作员发起。
- 后台可配置“需审核”或“自动通过”。
- 普通换卡申请通过后自动生成眼球卡记录。
- 补卡申请通过后不新建卡片记录，仅将关联卡片记录状态调整为待打印和待发。
- 申请被拒绝时，前台需显示拒绝理由。
- 用户可查看自己的申请记录与处理状态。

### 10.4 前台统计报表
功能要求：
- 游客可查看公开统计报表。
- 前台显示全部公开统计项，不限制时间范围。
- 报表内容不包含个人隐私信息。
- 报表仅在线查看，不提供导出功能。
- 报表形式为图表和数字数据，包含饼图等统计图形。
- 一期至少展示基础数量统计和按月份趋势统计。

## 11. 审计与日志要求

系统需提供审计和日志功能，至少包括：
- 通联记录的新增、修改、删除日志
- 卡片记录的新增、修改、删除日志
- 发信确认日志
- 收信确认日志
- 换卡申请审核日志
- 呼号绑定审核日志
- 导入导出任务日志
- 自动识别或自动填充操作日志
- 关键配置变更日志

建议支持的审计字段：
- 操作人
- 操作时间
- 操作对象
- 操作对象 ID
- 关联呼号
- 操作类型
- 关键字段变更前后值
- 操作结果

审计日志页面要求：
- 需要独立页面展示审计日志。
- 需要支持足够丰富的过滤。
- 至少支持按对象类型、对象 ID、关联呼号、操作人、操作类型、操作结果、时间范围等条件筛选。

## 12. 非功能需求

- 前台查询需具备基本防滥用能力。
- 后台关键操作需可追溯。
- 数据导入导出需支持备份恢复场景。
- 地址与个人信息需避免在公开页面泄露。
- 一期以功能稳定、数据准确、流程闭环为优先。

## 13. 当前已确认的核心业务规则

- 一条 QsoRecord 只允许对应一条 QslCardRecord。
- 眼球卡允许没有 QsoRecord。
- 收听卡必须关联 QsoRecord，并按正常通联卡规则处理。
- 同一呼号同一天多次通联，视为多个 QsoRecord，每条 QsoRecord 与一条 QslCardRecord 一一对应。
- 卡片状态机采用制作状态、寄出状态、收卡状态三段式设计。
- 通联记录和卡片记录去重采用既定组合键。
- 地址簿按“呼号 + 地址”去重。
- 游客可查询眼球卡、通联卡、收听卡相关公开记录，但不显示个人隐私信息。
- 游客仅查询呼号、卡片类型、寄出/收到状态与时间、必要通联时间。
- HAM 可查询自己呼号的详细记录（QsoRecord + QslCardRecord）。
- 前台查询需支持频次限制，默认每分钟 5 次。
- 游客按 IP 限流，登录用户按用户 ID 限流。
- Halo 原生用户体系作为账号基础。
- 一个用户可绑定多个呼号，且每个呼号都需单独审核。
- 呼号确认可通过证明截图或后台已记录手机号辅助核验。
- 手机号匹配多个呼号或多个地址时，也直接视为授权并自动通过。
- 呼号绑定审核与角色授权通过 Halo 用户管理功能实现，插件仅提供权限节点。
- 权限节点粒度精确到子菜单一级即可。
- 换卡申请支持审核模式与自动通过模式。
- 普通换卡申请仅用于眼球卡。
- 补卡申请必须关联已有卡片记录。
- 补卡申请仅允许 HAM 或操作员发起。
- 补卡通过后不新建卡片记录，仅将状态调整为待打印和待发，并清空原打印与寄出相关当前状态字段。
- 卡片记录仅保留 `补卡次数` 字段，默认值为 0。
- `补卡次数 = 0` 表示未发生补卡，`补卡次数 > 0` 表示实际补卡发信成功次数。
- 补卡次数在实际再次发信成功时加 1。
- 发信确认后自动补全已打印卡片和已打印信封状态，不允许例外。
- 不允许自动补建通联卡或收听卡，缺失时必须先手工创建后再执行收信确认。
- 删除采用逻辑删除并保留审计日志。
- 导入冲突规则为新数据覆盖旧数据、空值不覆盖非空值、人工修改数据允许被覆盖。
- 卡片导出和信封导出均采用 CSV，编码固定为 UTF-8 with BOM。
- 导出仅作为 Word 邮件合并的数据源，实际打印不在本项目开发范围内。
- 总览页面支持筛选结果导出 CSV；统计报表仅在线查看，不提供导出。
- 补卡判定间隔天数默认 7 天，且允许后台配置与启停。
- 后台必须具备审计和日志能力，且审计日志支持丰富筛选。
- 统计报表纳入一期，前台显示全部公开统计项且不限制时间范围，后台允许筛选且不预设默认范围。
- 报表形式为图表和数字数据。
- 二期功能在一期中不预留入口或占位。
- 插件需提供权限节点清单，实际授权通过 Halo 用户组配置完成。
- 项目采用重构重头开发方式，不继承旧系统实现。
