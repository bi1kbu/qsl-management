# 接口拉取与打印回写说明

本文说明如何让打印程序通过接口获取数据，并在打印结束后将状态回写到 QSL 插件。  
本版同时支持“全量读取、按需使用”。

## 1. 整体流程

1. 读取桥接配置文件（JSON）。
2. 调用远程导出接口获取 CSV 数据（卡片或信封，默认全量）。
3. 按映射规则把 CSV 字段转换为打印预设字段。
4. 执行打印任务。
5. 读取逐行打印结果，仅对 `success` 的行进行状态回写。
6. 回写结果统计（成功/失败/跳过）反馈到界面。

## 2. 必要接口

1. 拉取卡片 CSV：`POST /apis/qsl.admin/v1/exports/cards`
2. 拉取信封 CSV：`POST /apis/qsl.admin/v1/exports/envelopes`
3. 回写卡片状态：`PUT /apis/qsl.admin/v1/qsl-card-records/{id}`

鉴权可使用：
1. Basic（`Authorization: Basic ...`）
2. Bearer（`Authorization: Bearer ...`）

可选：`X-Operator` 作为操作人标识。

> 说明：当前导出接口在请求体为 `{}` 或无 `cardIds` 时，按全量导出处理。

## 3. 全量读取配置示例

```json
{
  "base_url": "http://localhost:8090",
  "auth": {
    "type": "basic",
    "username": "admin",
    "password": "admin",
    "operator": "admin"
  },
  "endpoints": {
    "cards": "/apis/qsl.admin/v1/exports/cards",
    "envelopes": "/apis/qsl.admin/v1/exports/envelopes"
  },
  "writeback": {
    "enabled": true,
    "datasets": {
      "cards": {
        "id_field": "卡片ID",
        "url": "/apis/qsl.admin/v1/qsl-card-records/{id}",
        "method": "PUT",
        "body": {
          "productionStatus": "PRINTED",
          "printedAt": "${now}"
        }
      },
      "envelopes": {
        "id_field": "卡片ID",
        "url": "/apis/qsl.admin/v1/qsl-card-records/{id}",
        "method": "PUT",
        "body": {
          "envelopePrinted": true,
          "envelopePrintedAt": "${now}"
        }
      }
    }
  },
  "filters": {
    "cards": {},
    "envelopes": {}
  },
  "mappings": {
    "cards": {
      "peerCallsign": "呼号",
      "Date": "日期",
      "Time": "时间"
    },
    "envelopes": {
      "peerCallsign": "呼号",
      "name": "姓名",
      "postcode": "邮编",
      "address": "收件地址"
    }
  }
}
```

## 4. 按需使用（推荐做法）

### 4.1 先全量拉取，再在打印端筛选

1. 保持 `filters.cards = {}`、`filters.envelopes = {}`，确保接口返回全量数据。
2. 在打印程序中按业务条件选择要打印的记录（例如呼号、状态、时间段）。
3. 仅把选中的记录加入打印队列，未选中记录不回写。

### 4.2 需要固定批次时，再启用配置筛选

如果你希望某次任务只拉取某类记录，可临时加 `filters`，示例：

```json
{
  "filters": {
    "cards": {
      "发出卡片": "0"
    },
    "envelopes": {}
  }
}
```

执行完该批次后建议恢复为空对象，避免下次漏数。
## 5. 字段映射规则

1. `mappings` 的 key 是打印预设字段 key。
2. 值可写源字段名（字符串），也可写对象规则（如 checkbox 转换）。
3. 程序会附加内部元数据（如数据集、记录 ID、配置路径）用于后续回写。

## 6. 回写规则

1. 仅回写打印结果为 `success` 的行。
2. 通过 `id_field` 从导出行中取卡片 ID。
3. `url` 支持 `{id}` 或 `${id}` 占位符。
4. `body` 支持模板变量：
   - `${id}`：卡片 ID
   - `${now}`：当前时间（秒级）
   - `${row.xxx}`：导出行字段
5. 回写失败不会阻断其他成功行回写，最终汇总失败详情。

## 7. 常见问题

1. 打印后界面仍显示“未打印”
   - 原因：回写字段不匹配。应回写 `productionStatus=PRINTED`，不是仅写 `cardPrinted=true`。
2. 信封已打印未生效
   - 检查是否回写 `envelopePrinted=true` 与 `envelopePrintedAt`。
3. 回写 401/403
   - 检查鉴权方式和账号权限。
4. 回写 404
   - 检查 `base_url` 与 `url` 前缀，确认插件接口路径正确。
5. 想“全量读取”但数据不全
   - 检查 `filters.cards` / `filters.envelopes` 是否被误配置了筛选条件。

## 8. 建议

1. 先在测试环境用单条记录验证“拉取 + 打印 + 回写”闭环。
2. 再做批量打印，观察回写统计中的失败明细。
3. 生产环境建议使用独立低权限账号作为回写账号。
4. 若需要“可追溯批次”，建议在打印前给队列打批次标签，并在回写 body 中附带 `${now}` 或批次号字段。
