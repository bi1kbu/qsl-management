# QSL 场景流程验证汇总

更新时间：2026-04-08  
环境：Halo `2.23.2`（开发容器 `halohub/halo:2.23.2`）

## 1. 后台录入 -> Python回写状态 -> 前台查询分阶段可见
- 场景目标：
  - 后台录入呼号、姓名、地址、电话、邮编、邮箱。
  - Python 获取信息并分步回写：卡片已打印、信封已打印、卡片已发出。
  - 前台 `QSL查询` 每一步状态可见。
- 结果：通过。
- 关键实现：
  - 前台状态新增阶段文案：`待打印卡片` -> `卡片已打印` -> `信封已打印` -> `本台已发卡`。
  - 公开查询返回补充字段：`envelopePrinted`、`envelopePrintedAt`。

## 2. 二维码收信确认（cardId + callsign）
- 场景目标：
  - 卡片含二维码，扫码访问收信确认 URL（带 `cardId`、`callsign` 参数）。
  - 自动填入后提交收信确认。
  - 前台可见“已签收”。
- 结果：通过。
- 关键实现：
  - 收信确认卡片支持 URL 参数自动填充：`cardId`、`callsign`、`remark`。
  - 呼号改为必填（前后端同时校验）。
  - 前台查询文案：`CONFIRMED` 显示为 `已签收`。

## 3. 后台按呼号收信（有记录更新 / 无记录补建）
- 场景目标：
  - 本台收到对方卡片后，后台流程改为：
    - 输入呼号 -> 搜索记录
    - 若有匹配记录：选择记录 -> 确认收信
    - 若无匹配记录：展开新增字段 -> 填写信息 -> 确认收信并新建
- 结果：通过。
- 关键实现：
  - 页面 `收信确认` 已按新流程改造为“先搜后分支”，新增字段仅在无匹配时显示。
  - `POST /apis/qsl.admin/v1/qsl-card-records/receive-confirm` 按呼号处理：
    - 有眼球卡：更新为 `returnCardStatus=RECEIVED`。
    - 无眼球卡：创建 `EYEBALL` 卡并直接 `returnCardStatus=RECEIVED`。
  - 补建必填字段：`name`、`address`、`postcode`、`phone`、`email`。
  - 缺字段时返回 `400`（不再是 500）。
  - Chrome MCP 验证结果：
    - 搜索 `BG9EXIST` -> 命中 1 条，勾选后确认收信，状态变更为 `RECEIVED`。
    - 搜索 `BG9NEWNOREC` -> 未命中，展开新增字段并提交后，自动新建且显示为可选记录。

## 4. 同地址多呼号封面合并导出 CSV
- 场景目标：
  - 多个呼号填写同一地址（卡局）时，导出信封 CSV 自动合并。
  - 收件人字段用多个呼号以顿号拼接。
- 结果：通过。
- 关键实现：
  - `POST /apis/qsl.admin/v1/exports/envelopes` 导出逻辑改为按地址聚合。
  - 合并后输出示例：
    - 呼号：`BG1AAA、BG2BBB`
    - 姓名：`BG1AAA、BG2BBB`
    - 收件地址：同地址一条

## 5. 换卡申请前台卡片（补充）
- 场景目标：
  - 前台提交换卡申请，自动进入眼球卡待发清单。
  - 已有眼球卡时阻止重复申请。
- 结果：通过。
- 关键实现：
  - 新增页面：`/plugins/qsl-management/widgets/exchange`。
  - 新增接口：`/plugins/qsl-management/widgets/public-api/actions/exchange-request`。
  - 重复拦截提示：`已交换过卡片，如需二次交换请联系人工处理`。
  - 卡片高度已做 iframe 自适应（无内部滚动）。

## 6. 页面参数对 iframe 卡片自动填充能力
- 结论：
  - 页面 URL 参数（如 `http://localhost:8090/1775055711015?cardId=...`）不会自动传给 iframe 卡片。
  - 原因：iframe 使用固定 `src`，不继承外层页面 query 参数。

## 7. 第三方插件 redirects 报错归因
- 现象：
  - 启用 redirects 插件时报 `IncompatibleClassChangeError`（`SettingFetcher.fetch`）。
- 结论：
  - 非 QSL 插件问题，为 redirects 插件与当前 Halo API 的兼容问题。
- 跟踪：
  - 已提 issue：<https://github.com/Leezj9671/halo-redirects/issues/2>

## 8. QSO 录入（后台录入完整字段）
- 场景目标：
  - 后台录入对方呼号、日期、时间、时区、频率、模式、设备、天线、功率、信号报告、备注。
- 结果：通过。
- 验证过程：
  - 在 `通联记录` 页面录入完整数据并提交。
  - 列表成功回显字段：`呼号/日期/时间/时区/频率/模式/设备ID/天线ID/功率ID/信号报告/备注`。
  - Chrome MCP 实测提交样例：
    - 呼号：`BG2MBC`
    - 日期时间：`2026-04-08 20:45:00`
    - 时区：`UTC+8`
    - 频率：`145.500MHz`
    - 模式：`FM`
    - 设备ID/天线ID/功率ID：`1001/1003/1005`
    - 信号报告：`59`
    - 备注：`场景：QSO完整录入验证`
- 关键实现：
  - `QsoView.vue` 表单新增字段：`timezone`、`equipmentId`、`antennaId`、`powerPresetId`、`rstSent`、`remark`。
  - 提交时对设备/天线/功率字段做数值化，避免字符串 ID 写入。
  - 页面加载时并发加载字典项（设备/天线/功率）和通联记录。
