# Agent 协作约定

## 适用范围
本约定作为仓库内 Halo2 插件工作的持久化记忆，默认长期生效。

## 强制流程（先设计后编码）
1. 先查资料：任何实现前，必须先查阅目标版本最新的 Halo2 官方文档与 API 变更记录。
2. 官方优先：技术决策优先采用官方推荐与最佳实践，避免临时性方案。
3. 先写基线：编码前先输出简要设计基线，至少覆盖：
   - 计划使用的 Halo2 接口与扩展点
   - 安全设计（认证、鉴权、攻击面）
   - 权限节点与 RBAC 角色模板映射
   - API 合同（group/version/path/method/认证要求）
4. 编码闸门：上述基线未明确且未达成一致前，不进入编码。
5. 编码后校验：完成实现后执行相关构建/测试，并再次核对与官方文档的一致性。

## Halo2 专项规则
1. 坚持最小权限原则：至少区分 `view` 与 `manage`，并定义权限依赖关系。
2. 服务端鉴权必须存在，不能仅依赖前端权限隐藏。
3. 新增 API 必须遵循 Halo2 规范（`/apis`、清晰的 API group/version、明确的 console/uc/public 范围）。
4. 任何偏离官方建议的实现，必须记录原因、风险与回退方案。

## 协作规则
1. 实施前需报告关键官方来源（链接 + 核验日期）。
2. 文档存在歧义时，先澄清歧义再继续实现。
3. 需求变化后，及时更新本文件，保持约定与当前实践一致。

## 版本兼容约定
1. 默认不做旧版兼容，所有新改动按当前版本目标直接实现。
2. 除非明确提出“需要兼容旧版”，否则不新增兼容层、兼容分支或兼容路由。
3. 发现存量旧版兼容代码时，应在本次相关改动中同步移除，并更新文档说明。

## 数据持久化约定
1. 后续所有功能默认必须支持持久化存储，不接受仅前端内存态或临时草稿态作为最终实现。
2. 若某功能阶段性只能提供非持久化能力，必须在评审前明确标注“临时方案”并给出切换到持久化的时间点。

## 语言约定
1. 所有注释、文档、页内标签/标题、面向用户的提示信息统一使用中文。
2. 若第三方接口强制英文（如协议字段、API 必需字面量、框架保留键），保留必需英文标记，并尽量在附近提供中文说明。
