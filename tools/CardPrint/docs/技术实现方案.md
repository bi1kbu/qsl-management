# CardPrint 技术实现方案

## 1. 目标与约束

1. 支持 Windows 针式打印机直接打印，不依赖先转 PDF/图片。  
2. 预览坐标系固定为“用户视角左上角原点”，打印时根据设备配置做旋转与偏移映射。  
3. 标定 UI 与打印 UI 均通过调用 CLI 执行打印相关动作。  
4. 保证“同一份预设 = 同一预览结果 = 同一打印结果”。  

## 2. 技术选型

1. Python `3.12+`。  
2. CLI：`Typer`（命令组织清晰，参数提示好）。  
3. 打印机访问：`pywin32`（`win32print`, `win32ui`, `win32con`, `win32gui`）。  
4. UI：`PySide6`（原生桌面体验，绘制/预览能力稳定）。  
5. 数据模型与校验：`pydantic v2`。  
6. 表格导入：`pandas + openpyxl`。  
7. 测试：`pytest`。  

## 3. 总体架构

```
UI(标定) ──调用──> CLI ──调用──> Core Engine ──> Win32 Printer Adapter
UI(打印) ──调用──> CLI ──调用──> Core Engine ──> Win32 Printer Adapter
                                 └──> Preview Engine (共享坐标/布局逻辑)
```

设计原则：

1. 旋转、原点、死区、字段布局全部在 `Core Engine` 中统一实现。  
2. CLI 与 UI 只做编排，不重复实现坐标和绘制规则。  
3. UI 调 CLI 使用 `subprocess` + JSON 输入/输出，保持边界明确。

## 5. 关键数据模型

单位统一使用 `mm` 存储，渲染和打印时再转换为设备单位。

### 5.1 预设文件 `preset.json`

```json
{
  "version": "1.0",
  "name": "医保卡-模板A",
  "paper": {
    "name": "CustomCard",
    "width_mm": 85.6,
    "height_mm": 54.0
  },
  "calibration": {
    "rotation_direction": "right",
    "rotation_degree": 90,
    "origin_offset_mm": { "x": 2.1, "y": 1.3 },
    "deadzone_mm": { "top": 1.0, "right": 1.2, "bottom": 1.1, "left": 0.8 }

  "fields": [
    {
      "key": "name",

      "font_family": "SimSun",
```

## 6. 坐标与变换策略（关键）

逻辑坐标系：

1. 原点固定左上角。  
2. 所有字段在该坐标系编辑和预览。  
3. 打印前才执行旋转 + 原点偏移映射。  

推荐实现：矩阵变换链（2D 仿射）

```
physical_point = T(origin_offset) * R(rotation, paper_size) * logical_point
```

规则：

1. `R` 仅由 `rotation_direction + rotation_degree` 决定。  
2. `T` 由标定实测得到。  
3. 在逻辑坐标放置字段时，先检查是否落入死区；若落入则 UI 拒绝保存。  
4. 预览与打印都调用同一 `transform.py`，禁止复制公式。  

## 7. CLI 设计（UI 调用入口）

命令分组建议：

1. `cardprint printer list`  
   - 输出可用打印机（JSON）。  
2. `cardprint printer papers --printer <name>`  
   - 输出该打印机可用纸张。  
3. `cardprint calibrate print-cross --printer ... --paper ...`  
   - 打印标定十字与边距标尺。  
4. `cardprint preset validate --preset <path>`  
   - 校验预设完整性与死区冲突。  
5. `cardprint render preview --preset <path> --row <json>`  
   - 输出预览元素 JSON（供 UI 绘制，不直接耦合 UI 组件）。  
6. `cardprint print run --job <job.json>`  
   - 执行正式打印，返回每条记录成功/失败。  

输出规范：

1. 标准输出仅 JSON。  
2. 错误输出统一结构：`code/message/details`。  
3. 进程返回码：`0` 成功，非 `0` 失败。  

## 8. 标定 UI 设计

页面结构：左侧参数与步骤、右侧预览。

步骤流：

1. 选择打印机与纸张（调用 CLI 枚举）。  
2. 打印标定页（十字+边距刻度）。  
3. 输入实测值（十字到上边距/左边距）。  
4. 计算 `origin_offset_mm` 并实时更新预览。  
5. 输入死区值，预览区显示红色半透明遮罩。  
6. 编辑字段模板与字体样式。  
7. 保存预设 `preset.json`。  

## 9. 打印 UI 设计

页面结构：左侧输入与数据源、右侧预览与打印队列。

流程：

1. 载入预设，标定参数只读显示。  
2. 字段值输入方式：手工、CSV、XLSX。  
3. 每行数据可切换预览。  
4. 点击打印后生成 `job.json` 调 CLI 执行。  
5. 显示逐行打印结果与失败原因。  

## 10. 数据源插件预留

先定义接口，不在第一阶段实现联网细节。

```python
class JsonDataSourcePlugin(Protocol):
    def name(self) -> str: ...
    def fetch(self, query: dict) -> list[dict]: ...
```

约束：

1. 插件输出字段必须与 `preset.fields[*].key` 对齐。  
2. 插件异常不影响主流程，UI 给出可读错误。  

## 11. 一致性与质量保障

### 11.1 一致性机制

1. 预览和打印共用 `layout_engine + transform`。  
2. CLI `render preview` 与 `print run` 调用同一布局结果（一个用于显示，一个用于落笔）。  
3. 每个预设保存时做死区与越界校验。  

### 11.2 测试策略

1. 单元测试：
   - 旋转矩阵与坐标映射（0/90/180/270）。
   - 死区冲突检测。
   - 预设 JSON schema 校验。
2. 集成测试：
   - CLI 命令出参结构。
   - 使用测试打印机驱动验证调用链。
3. 回归测试：
   - 固定预设 + 固定输入 -> 固定逻辑布局快照（JSON 快照）。

## 12. 实施顺序

1. Phase 1：初始化工程、数据模型、CLI 骨架、Win32 打印基础。  
2. Phase 2：坐标变换与死区校验、标定 CLI、预设保存。  
3. Phase 3：标定 UI（可完整生成预设）。  
4. Phase 4：打印 UI（手工输入 + CSV/XLSX）。  
5. Phase 5：联调、测试、文档、发布。  

## 13. 风险与规避

1. 不同打印机驱动坐标行为差异：通过“标定页 + 原点偏移 + 死区”统一吸收。  
2. 字体在驱动中的实际渲染偏差：预设中记录字体家族与大小，增加字体可用性检查。  
3. UI 和 CLI 逻辑分叉风险：严格禁止 UI 自己计算坐标，只消费 CLI/核心引擎结果。
